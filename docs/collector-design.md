## 1. 왜 필요한가

### 1.1 지금 상태

수집기 자체는 돈다. 오늘 맥에서 돌려 **경기 2,200 노선 풀 + 관측 23,517건**을 모았다. 문제는 **운영**이다.

| 지금 | 문제 |
|---|---|
| 맥에서 `launchd` 로 실행 | **맥이 꺼지면 멈춘다.** 노트북이면 덮는 순간 끝 |
| 상태를 CLI 로만 조회 | 다른 컴퓨터에서 무인 운영하면 **볼 방법이 없다** |
| 완료 시점을 모름 | 진행률이 있어야 "언제 끝나나"를 판단 |

### 1.2 ★ 결정적 이유 — 밴드 구멍은 시간이 해결하지 못한다

**컴퓨터가 꺼진 시간대의 데이터는 며칠을 기다려도 0이다.** 무작위 손실이 아니라 **구조적 구멍**이다.

✅ 오늘 10:24 부터 돌린 실측:

```
밴드0   4-7 시  관측      0
밴드1   7-9 시  관측      0    ← 출근 첨두
밴드2   9-12시  관측 18,259  ████████████████████████████████
밴드3  12-15시  관측      0
밴드4  15-17시  관측      0
밴드5  17-20시  관측      0    ← 퇴근 첨두
밴드6 20-03시  관측      0
```

**켜둔 밴드 하나만 찬다.** 그리고 §4.4 의 "경기 10 평일" 산수는 **하루 종일 돈다는 가정** 위에 있다. 하루 6시간만 켜면 3배 걸리는 게 아니라 **나머지 밴드가 영원히 안 찬다.**

노트북이면 특히 나쁘다 — **7-9시는 출근길, 17-20시는 퇴근길에 덮여 있다.** 차별점이 제일 큰 두 밴드가 정확히 그때다.

→ **항상 켜진 기계가 필수다.** 이 프로그램은 그 기계에서 무인으로 돌고, 원격에서 상태를 볼 수 있어야 한다.

## 2. 오늘 비싸게 배운 것 — 설계에 반드시 반영할 것

전부 실제로 데이터를 잃고 나서 알아낸 것들이다. **다시 밟지 않도록 프로그램이 구조적으로 막아야 한다.**

### 2.1 실패가 HTTP 200 으로 온다 ⚠️ 최우선

```json
{"response":{"header":{"resultCode":99,"resultMsg":"가용한 세션이 존재하지 않습니다. (30/30)"},"body":""}}
```

**HTTP 200 이고 예외도 안 난다.** `resultCode` 를 안 보면 **"버스가 없다"로 읽힌다.** 실제로 이 함정에 빠져 **요청의 2/3를 모르게 버렸다.**

**→ 요구사항:** `resultCode` 를 항상 검사한다. **실패는 절대 조용히 넘어가지 않는다.** 실패율을 지표로 노출하고 임계치를 넘으면 경고한다.

### 2.2 rate 가 아니라 버스트가 문제다

✅ 같은 평균 rate, 정반대 결과:

| 방식 | 결과 |
|---|---|
| **균등** 6.0/s (167ms 간격 제출) | **88건 전부 성공, 429 0건** |
| **버스트** 6.7/s (20개 동시 투입) | **180건 중 140건 실패 (78%)** |

`ThreadPoolExecutor.map` 은 워커 수만큼 한꺼번에 던진다 → 토큰 버킷에 걸려 `HTTP 429 API token rate limit exceeded`.

✅ 균등 제출 실측: 2·4·6/s = 무결점, 10/s 부터 141/144 로 섞이기 시작.

**→ 요구사항:** 제출을 `1/rate` 간격으로 벌린다. rate 는 설정값으로 노출한다.

### 2.3 제약이 두 개고 성격이 다르다

활용가이드의 `30 TPS · 평균 응답 500ms` 는 **둘 다 틀렸다.**

| 활용가이드 | 실측 |
|---|---|
| 평균 응답 500ms | **~3초** (6배) |
| 30 TPS 하나 | **동시 세션 30개** + **토큰 rate limit** — 별개의 두 제약 |

**→ 요구사항:** 여러 프로세스를 동시에 돌릴 때 **in-flight 합이 30을 넘지 않게** 한다. (오늘 `fetch_routes`(18) + 수집기(18) = 36 세션으로 서로를 죽였다.)

### 2.4 재시작이 쿼터 카운터를 리셋하면 안 된다

`KeepAlive` 로 재시작되면 메모리 카운터는 0이 된다. 절전 복귀·크래시가 몇 번 나면 **일 상한을 넘겨 그날 수집이 통째로 죽는다.**

**→ 요구사항:** 일 호출수는 **디스크에 원자적으로** 기록한다. (`os.replace`)

### 2.5 중복 폴링을 하지 않는다

이미 목표를 채운 구간을 계속 재폴링하면 그 콜로 **아직 한 번도 안 본 노선**을 볼 수 있었다. 총 소요가 **달력이 아니라 커버리지에 묶여야** 한다.

**→ 요구사항:** `(노선, 구간, 밴드, 요일)` 셀별 샘플 수를 세고, 채운 노선은 폴링 목록에서 빼고 미커버를 넣는다.

### 2.6 ★ 안 도는 노선을 찍지 않는다 — 커버리지만 보면 안 된다

커버리지 순으로만 고르면 **지금 실제로 운행하는 노선인지를 안 본다.** 심야에 "미커버 순 N개"를 고르면 **이미 차고에 들어간 낮 노선**을 찍고, 정작 **아직 달리는 노선을 버린다.**

✅ 실측 — 시각별로 운행 중인 노선 수가 20배 넘게 변한다:

```
00:30   462 노선      08:30  1,775 노선
01:30   116           12:30  1,764
02:30    97           20:30  1,548
03:30    86           23:30    981
04:30   239
```

**"심야엔 40개"같은 상수를 박으면 01시에 76개를 버린다.** 그리고 하필 그 시간에 도는 게 `36`번(05:16 도착)·`9709N` 같은 **광역/직행좌석**인데, §6.3 이 *"버스에서 차별점이 보이는 유일한 케이스"*로 지목한 부류가 정확히 그것이다.

**→ 요구사항: 상수를 쓰지 말고 두 단계로 거른다.**

| 단계 | 근거 | 역할 |
|---|---|---|
| ① 운행시간 창 | `startvehicletime`/`endvehicletime` (전수 확보) | **명백히 아닌 것만** 제외 |
| ② `emptyStreak` | **관측** — 연속 0대면 후순위 | 애매한 것은 폴링이 정한다 |

⚠️ **①만으로는 부족하다.** `endvehicletime` 은 막차 **출발** 시각이고, 그 뒤로 종점까지 얼마나 더 달리는지는 **어느 API 도 주지 않는다**(§4.4 의 그 결측). 그래서 창을 넉넉히(`tail_min`) 열고 **실제로 비었는지는 폴링해서 안다.** 추정으로 못 박으면 데이터를 잃는다.

**부수 효과 — 낮에도 이득:** 05:30 에 944개만 운행 중이므로 나머지 1,256개(첫차 전)를 안 찍는다. 빈 응답에 콜을 안 태운다.

### 2.7 자정을 넘기는 처리 — 조용히 데이터를 버린 버그 둘

| 버그 | 증상 |
|---|---|
| `band_of()` 가 `t.hour` 만 봄 | 밴드 `[20,24]` 에 **00:30 이 안 걸린다** → 어느 밴드에도 없으니 **커버리지에 영원히 안 들어감** |
| `day_type()` 이 벽시계 기준 | **토요일 01:30 버스는 금요일 막차**인데 '토요일' 표본으로 셌다 |

**→ 요구사항:** 밴드·운행창의 끝값이 24 를 넘으면 익일로 해석한다(`[20,27]` = 20:00~03:00). 요일은 **운행일**(04시 기준) 로 판정한다.

## 3. 구성

```
┌─ 관리 프로그램 ────────────────────────────────┐
│                                                │
│  수집기(들)          조율기            대시보드 │
│  bus / subway   →   커버리지 DB   →   웹 UI    │
│       ↓              노선 선택          ↑      │
│    JSONL(원본)        쿼터 가드      원격 조회  │
└────────────────────────────────────────────────┘
```

| 구성요소 | 있나 | 역할 |
|---|---|---|
| `bus_collector.py` | ✅ 있음 | 위치 폴링 → 정류장 통과 기록 |
| `subway_collector.py` | ✅ 있음 | 열차 위치 → 정시성 검증용 |
| `orchestrator.py` | ✅ 있음 | 커버리지 DB(SQLite) + 노선 선택 + `status` |
| `fetch_routes.py` | ✅ 있음 | 노선 풀 (커버리지 분모) |
| **`server.py`** | ✅ **만듦** | 수집기 + 대시보드를 한 프로세스로. 표준 라이브러리만 |
| **운영 설정** | ❌ 없음 | 자동시작 · 생존 · 배포 — **대상 기계가 정해져야 함** |

```bash
python3 server.py              # 수집 + 대시보드 (기본 877)
python3 server.py --port 9000
python3 server.py --no-collect # 대시보드만 (조회용)
```

## 4. 대시보드 — 무엇을 보여줄 것인가

**원칙: 오늘 우리를 속인 것들이 화면에 보여야 한다.** 조용한 실패, 밴드 구멍, 쿼터 소진.

### 4.1 완성률 (핵심)

```
전체    ████████░░░░░░░░░░░░░░░░  32.1%     (328,912 / 1,024,534 셀)
예상 완료  2026-07-30 (7 평일 남음)
```

- 분모 = `구간수 × 밴드수` (평일 기준). ✅ 경기 = 146,362 구간 × 7밴드 = **1,024,534 셀**
- 셀이 `targetSamples` 를 채우면 완료
- 예상 완료 = 남은 셀 ÷ 최근 실측 관측률 (가정값이 아니라 **실제 rate 로 역산**)

### 4.2 ★ 밴드별 — §1.2 의 구멍이 보여야 한다

```
밴드0   4-7 시   ████████████████░░░░  78%   ← 채워지는 중
밴드1   7-9 시   ██████████████████░░  91%   첨두
밴드2   9-12시   ████████████████████ 100%  ✅
밴드3  12-15시   ███████████████░░░░░  74%
밴드4  15-17시   ░░░░░░░░░░░░░░░░░░░░   0%  ⚠️ 24시간 이상 관측 없음
밴드5  17-20시   ██████████████░░░░░░  68%   첨두
밴드6  20-03시   ███████████░░░░░░░░░  55%   익일
```

**어떤 밴드가 안 차고 있는지가 한눈에 보여야 한다.** 그게 컴퓨터가 꺼져 있었다는 유일한 신호다.
→ **일정 시간 이상 관측이 없는 밴드는 경고로 띄운다.**

### 4.3 요일별 — 토·일 병목

```
평일   ████████░░░░░░░░  32%
토요일 █░░░░░░░░░░░░░░░   4%   ⚠️ 주 1일씩만 얻음 — 목표 7샘플이면 7주
일요일 █░░░░░░░░░░░░░░░   4%   ⚠️ 동일
```

⚠️ **토·일은 커버리지 전략으로 못 줄인다.** 주에 1일씩만 오기 때문이다. 대시보드는 이걸 **평일과 분리해서** 보여야 하고, 평일 진행률에 섞으면 안 된다.

### 4.4 건강 상태 — 조용한 실패를 잡는다

```
키별 쿼터 GBIS_BUS_KEY 231,405 / 480,000 · KEY2 229,881 / 480,000
실패율    1.7%   (code99 ×0 · HTTP429 ×3 · timeout ×0)     ✅
사이클    30.4s   (목표 30s)
관측률    4.4건/초
마지막 관측  12초 전
```

- **실패를 유형별로 쪼개서** 보여준다 — `code99`(세션 초과)와 `429`(rate)는 대응이 다르다
- 실패율이 임계치(예: 5%)를 넘으면 경고
- **"마지막 관측"이 멈추면 죽은 것** — 이게 제일 중요한 생존 신호

### 4.5 노선/시군별 진행

- 시군별 완성률 (경기 31개 시군)
- 진행 중 노선 목록 (상한 340개 — 운행 중인 게 적으면 그만큼만)
- 완주한 노선 수

### 4.6 형태

| 안 | 장단 |
|---|---|
| **로컬 웹 대시보드** (stdlib `http.server`) | 원격 조회 가능. 의존성 0. **권장** |
| CLI (`orchestrator.py status`) | 이미 있음. 원격에선 SSH 필요 |
| 상태 파일 + 동기화 | 단순하지만 실시간성 없음 |

**웹을 권장한다.** 다른 컴퓨터에서 무인으로 도니 브라우저로 열어볼 수 있어야 한다. 파이썬 표준 라이브러리만으로 충분하다 (현재 수집기도 stdlib 만 쓴다).

> ⚠️ **인증을 붙일지 정해야 한다.** 집 안 네트워크면 불필요할 수 있고, 외부에 열면 필요하다. **API 키가 같은 기계에 있으므로** 대시보드를 외부에 노출할 거면 신중해야 한다.

## 5. 운영

### 5.1 대상 기계

**항상 켜져 있어야 한다** (§1.2). 부하는 아주 가볍다:

| | |
|---|---|
| 요청 | **4.7건/초** — I/O 바운드, CPU 거의 안 씀 |
| 저장 | 하루 ~50MB × 14일 ≈ **700MB** |
| 의존성 | **파이썬 표준 라이브러리만** |
| 가동 | **24시간** — ✅ 전수 실측상 버스가 0인 시간대가 없다(§2.6) |

→ 라즈베리파이든 작은 VM 이든 충분하다.

### 5.2 ⚠️ 미확인 — 확인하고 진행할 것

| 항목 | 왜 중요한가 |
|---|---|
| **`data.go.kr` 이 해외 IP 를 막는가** | 막으면 국내 리전이어야 한다. **클라우드로 갈 거면 선결** |
| **키를 서버에 어떻게 둘 것인가** | 지금은 `.env`(600). 서버에선 관리 방식을 정해야 함 |
| 자동시작 방식 | macOS = `launchd` / Linux = `systemd`. 대상 기계에 따라 |

### 5.3 macOS 라면 — TCC 함정

`~/Desktop`·`~/Documents`·`~/Downloads` 는 **TCC 보호 폴더**다. 터미널에서 직접 실행하면 되지만 **`launchd` 로 뜬 프로세스는 접근 권한을 물려받지 못한다:**

```
python3: can't open file '.../Desktop/.../bus_collector.py': [Errno 1] Operation not permitted
```

→ **보호 폴더 밖에 둔다.** `python3` 에 전체 디스크 접근 권한을 주는 방법도 있으나 범위가 너무 넓다. (오늘 `~/findpath-collector/` 로 옮겨 해결했다.)

### 5.4 크래시 대응

`KeepAlive`(launchd) / `Restart=always`(systemd) 로 자동 재시작하되, **§2.4 대로 쿼터 카운터가 디스크에 있어야** 재시작이 상한을 넘기지 않는다.

> ✅ 오늘 실제로 크래시가 났다 — 사이클이 `intervalSec` 을 넘으면 `sleep` 인자가 음수가 되어 죽었다. **자동 재시작이 없으면 그대로 멈춘다.**

## 6. 설정 (`config.json`)

이미 있고, **각 값이 무엇을 지배하는지 주석으로 달려 있다.** 대시보드에서 읽기 전용으로 노출한다.

| 키 | 현재 | 지배하는 것 |
|---|---|---|
| `cityCodes` | 경기 31개 시군 (2,200 노선) | 범위. 서울은 별도 API(§9 #5) |
| `targetSamples` | 7 | **"언제 멈추나"만 정한다** — 수집 동작은 같으므로 나중에 바꿔도 된다 |
| `timebands` | `[4,7] [7,9] [9,12] [12,15] [15,17] [17,20] [20,28]` | ⚠️ **수집 기간을 지배한다.** 19개면 41 평일, 7개면 10 평일. `28` = 익일 04시(운행일 경계) — 27로 두면 03시대 첫차·막차가 장부 밖 |
| `serviceWindow` | **`[0,24]` 24시간** | ✅ 전수 실측: **버스가 0인 시간대는 없다** → §2.6 |
| `dispatchRate` | 6/s | ⚠️ **사이클 = 노선수 ÷ rate.** §2.2 |
| `maxRoutes` | **340** | 두 키의 독립 세션 풀에 분산하는 상한 |
| `maxWorkers` | 25 | in-flight 그릇. §2.3 의 세션 30 상한 주의 |
| `busKeyDailyCap` | 480,000 | 각 키의 50만 하드리밋에 독립 마진 |

> **`nightRoutes` 같은 상수는 두지 않는다** — §2.6 대로 운행시간 필터 + `emptyStreak` 가 정한다.

**밴드 × 샘플이 기간을 지배한다** (§4.4):

| 밴드 × 샘플 | 경기 2,200 |
|---|---|
| 19 × 10 | 41 평일 |
| **7 × 7 (현재)** | **10.4 평일 ≈ 2주** |
| 4 × 10 | 8.7 평일 |

⚠️ **밴드를 줄이는 쪽이 위험하다** — 07-09시와 09-12시 주행시간이 2배 차이 날 수 있는데 뭉치면 그 차이가 잡음이 된다. **밴드를 지키고 샘플을 줄일 것.**

### 7.2 기록 형식

> **⚠️ 추기:** 현행은 양 끝 `nodeid`와 `route_version`을 저장하고 정류장명·좌표는
> `route_stop` 버전 스냅샷으로 조인한다. `daytype`은 7종(mon~sun)이다.
> 인접·동일 밴드·동일 운행일 직접 전이만 정확한 셀에 넣고 건너뜀/경계 전이는
> `span`에 분리한다. 자세한 스펙은 [`../collector/README.md`](../collector/README.md) 참조.

**버스** — 정류장을 넘은 순간만. 통과 시각은 `(t_prev, t]` 안에 있다:

```json
{"t":"...","t_prev":"...","routeid":"GGB...","routeno":"9000","routetp":"직행좌석버스",
 "cityCode":31020,"vehicleno":"경기77바3714","from_ord":31,"to_ord":32,
 "nodeid":"...","nodenm":"순천향대학병원","band":2,"daytype":"weekday"}
```

⚠️ **API 에 타임스탬프 필드가 없다.** 폴링 시각이 곧 관측 시각이고 그만큼 오차가 얹힌다.
✅ 실측: 30초 폴링이면 이동의 **94.7%를 1칸**으로 잡는다 (2칸 이상 건너뜀 5.2%). 오차 중앙 37초지만 **통계 모델이라 `37s/√N` 로 상쇄**된다 — 개별 예측이 아니라 밴드별 평균을 만들기 때문.

**⚠️ 이 API 의 해상도는 정류장 단위다.** `gpslati`/`gpslong` 은 버스 GPS 가 아니라 **"현재 정류장의 좌표"**이고 정류장을 넘을 때만 바뀐다 (✅ 28초 동안 8대 전부 좌표 불변). 카카오가 *"기존 도착정보는 10~20초 갱신이라 실제 주행 상황을 반영 못 한다"*며 **고정밀 위치를 직접 연동**한 이유다. **그러나 우리 목적엔 이 해상도로 충분하다** — 필요한 건 `stop_times`(정류장 통과 시각)이지 정류장 사이의 위치가 아니다.

## 8. 이 데이터는 무엇을 위한 것인가

**⚠️ 지금 채택된 아키텍처(B+)에는 필요 없다.** B+ 는 TMAP `sectionTime` 을 쓴다.

**C(자체 GTFS + OTP)를 되살릴 선택지를 위한 수집이다.** 데이터는 기다리면 오지만, C 의 나머지 작업(GTFS 조립 · OTP 구동 · 커스텀 계산기 · OSM 매칭)은 안 온다 — 그래서 **수집만 미리 돌려두고 판단은 나중에** 한다.

**전환 설계:** B+ 는 **남의 시각을 안 쓴다**(`routeId` 와 노선 순서만 받아 시각은 자체 전파). 그래서 갈아끼울 자리가 하나로 좁혀진다:

```
경로 후보 생성  ←  [TMAP] 또는 [OTP]     ★ 여기만 교체
     ↓  (노선 순서만. 시각 없음)
시각 전파 · 도보 · 신호 · UI  ←  전부 공통
```

→ **2주 뒤 §9 의사결정 지점에서 두 제공자를 같은 요청에 태워 A/B 로 비교**하면, "신호를 비용함수에 넣으면 순위가 실제로 뒤집히는가"를 추측이 아니라 실측으로 알 수 있다.
