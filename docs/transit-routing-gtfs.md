# 도보 + 대중교통 길찾기 — 설계

> **범위:** 전국 목표 / **현 단계 = 서울·경기 전역** (§1.0)
> **차별점:** 걷는 속도 + 보행신호 대기를 소요시간에 반영
> **아키텍처:** **B+ — TMAP 뼈대 + 자체 시각 전파** (§2). ~~C(자체 GTFS + OTP)~~ 는 §4.4 쿼터 산수로 기각·보류
> **최종 갱신:** 2026-07-16 — 실측으로 다수 정정됨. ✅ 표시가 실측, 없으면 추정이다.

## 1. 목표와 가설

### 1.0 서비스 정의와 범위

**만드는 것: 도보 + 대중교통 길찾기 서비스.** 네이버·카카오지도의 길찾기와 같은 성격이되 **자동차는 제외**하고 도보·대중교통만 다룬다.

**차별점:** 걷는 속도(사용자 설정)와 **보행신호 대기**를 소요시간에 반영한다. 기존 서비스가 하지 않는 부분이며, 이 문서 전체가 그것이 유의미한지를 따진다.

**서비스 범위**

| | 범위 |
|---|---|
| 최종 목표 | **전국** |
| **현 단계 = 최소 요건** | **서울 · 경기 전역** |
| 검증 시나리오 | 판교 (성남시 `31020`) |

> **⚠️ 지역별 전략이 다르다 (§2.3.1)** — 쿼터가 자릿수로 달라서 같은 방식을 못 쓴다:
> | | 버스 | 지하철 |
> |---|---|---|
> | **경기** | **수집 + 조회** (운영 50만/일) | — |
> | **서울** | **조회만** (운영 1만/일 — 수집은 152배 부족) | ✅ 파일 + `realtimePosition` |

> ### ⚠️ "서울+경기"가 최소 요건이다 — 규모 산정의 기준선
>
> **판교 한 시나리오가 도는 것은 검증 수단이지 제품이 아니다.** 쿼터·비용·데이터 조달·일정은 전부 **서울+경기 전역**을 분모로 계산해야 하며, **단일 노선 예시로 규모를 가늠하면 안 된다.**
>
> | 대상 | 노선 수 |
> |---|---|
> | 경기 시내버스 | **2,200** (✅ 실측 — `getRouteNoList` 31개 시군 `totalCount` 합) |
> | 서울 시내버스 | 미실측 (키 미등록 §3.7). 통상 ~375로 알려짐 |
> | **합계** | **약 2,500~2,600** |
>
> 이 숫자가 §4.4 폴링 쿼터 산수의 입력이며, **C 아키텍처를 기각시킨 근거**다. bus-test의 311번은 도보·신호 모델을 확인하기 위한 **고정 시나리오일 뿐**이고, 그 규모로 계산하면 답이 두 자릿수 배수로 틀린다.
>
> 전국 확대는 서울+경기가 선 뒤에 판단한다 (§9).

### 1.1 가설

**가설:** 기존 길찾기(카카오·네이버·TMAP)는 **도보 구간에 신호 대기가 0초**로 들어가 있어 소요시간이 체계적으로 낙관적이다. 이를 반영하면 유의미하게 다른 답이 나온다.

**근거 (✅ 실측):** TMAP 보행자 API 응답의 구간별 속도를 역산하면

| 구간 유형 (`facilityType`) | 실측 속도 |
|---|---|
| 일반보행로 (11) | 1.20 m/s |
| **횡단보도 (15)** | **1.52 m/s** |
| 계단 (17) | 1.05 m/s |
| 지하보도 (14) | 0.93 m/s |

**횡단보도가 일반 보행로보다 빠르다.** 신호 대기가 조금이라도 반영됐다면 훨씬 느려야 한다. 즉 대기 = 0초.

**차별점이 성립하는 조건:** 신호 대기(약 90초)가 **바닥 오차보다 커야** 측정된다. 바닥 오차 10분짜리 시스템에 90초를 넣는 건 무의미하다 → **정확도 확보가 신호 반영보다 선행**한다.

## 2. 아키텍처 — TMAP 뼈대 + 자체 시각 전파 (B+)

> ### 🔄 결정 변경: C → B+ (2026-07-16)
>
> 이전 판은 **C(자체 GTFS + OTP)**로 결정했다. **두 가지가 그 결정을 뒤집었다:**
>
> 1. **일정** — C 는 `stop_times` 수집이 선행되어야 하는데(§4.4) 그게 **경기만 약 2주**다. 구현 예산이 **2주 전체**이므로 수집이 끝나는 시점이 곧 마감이다. 그 위에 GTFS 조립 · OTP 구동 · 커스텀 계산기(§7.2) · 표준데이터 매칭(§7.1) 이 얹힌다.
> 2. **C를 고른 근거 #1이 사실이 아니었다** — 아래 참조.
>
> ⚠️ **이전 판의 "7개월"은 지어낸 숫자였다.** 근거 없는 "3주/노선"에서 나왔고 실측하니 **1.0~1.4일**이다(§4.4). **5배 틀렸다.** 결론은 같은 자리로 왔지만 **"14배 차이라 논외"가 아니라 "빠듯해서 안 맞는다"** 가 정확한 이유다. → **C 재검토는 7개월 뒤가 아니라 2주 뒤다** (§9).

### 2.1 네 가지 안

| | 경로 선택 | 승차 시각 | 도보 시각 | 신호 반영 | 일정 |
|---|---|---|---|---|---|
| **A.** 남의 API 그대로 | TMAP | TMAP | TMAP | ❌ 불가 | 며칠 |
| **B.** A + 도보만 자체 보정 | TMAP | **TMAP `totalTime` + 도보 델타** | 자체 | △ 첫 도보만 | 1주 |
| **B+.** B + 자체 시각표로 시각 전파 | TMAP (**N개 후보 재정렬**) | **자체 시각표** | 자체 | ✅ **도보 전 구간 + 절벽** | **2주** |
| **C.** 전부 자체 | OTP | OTP | OTP | ✅ **비용함수에** | **수집 2주 + 조립·구동** |

#### A. 남의 API를 그대로 쓴다

TMAP/카카오/네이버 응답을 그대로 표시한다. **도보 속도도 신호도 반영할 자리가 없다** — 우리가 계산하는 게 없으므로. §1 가설을 검증할 수단 자체가 없어 제품이 성립하지 않는다.

#### B. 도보 구간만 자체 보정한다

TMAP 경로를 받고, 도보 구간만 우리 모델로 다시 계산해 `totalTime`에 델타를 더한다.

```
총시간 = TMAP totalTime + Σ(우리 도보시간 − TMAP 도보시간)
```

**❌ 이 방식은 조용히 틀린다.** `totalTime + 델타`는 **"도보가 밀려도 같은 차를 탄다"**는 암묵 가정이다. 실제로는 그 차를 놓치고 다음 차를 타므로 총시간이 배차간격만큼 점프한다. **정확히 그 점프가 우리가 재려는 것인데, B는 구조적으로 그걸 못 본다.**

`searchDttm`은 **출발 시각 하나만** 미는 파라미터라 중간 환승 도보는 보정할 방법이 없다. → **첫 도보만 부분 반영.**

#### B+. 경로 뼈대만 받고 시각은 우리가 전파한다 ★ 채택

TMAP에게서 **"어느 노선을 어느 순서로 타는가"**만 받고, **시각은 자체 시각표로 처음부터 다시 쌓는다.** TMAP의 `totalTime`·`sectionTime`은 **지하철 구간에서 아예 쓰지 않는다.**

```
08:00 출발
  → 첫 도보 420m : TMAP 6분 → 우리 7분 30초 (속도 3.5km/h + 신호 60초)
  → 역 도착      : 08:07:30
  → 자체 시각표 조회 → 08:07 열차 ❌ 놓침 → 08:12 열차 탑승   ★ 절벽
  → 하차 시각    : 시각표 (TMAP sectionTime 아님)
  → 환승 도보    : 우리 계산 → 도착 시각
  → 자체 시각표 조회 → 다음 열차                              ★ 절벽
  → 도착
```

**도보 90초가 총 5분이 된다.** B가 못 보는 그 점프가 여기서 나온다. 중간 환승도 절대 시각을 들고 있으므로 몇 번이든 반복된다 — `searchDttm` 제약과 무관하다.

**버스 구간은 다르다.** 시각표가 없으므로(배차간격 운영) TMAP `sectionTime` + 기대 대기를 쓰고, **절벽은 발생하지 않는다** — "원래 탈 그 차"가 없다(§6.3). 도보 델타는 정확히 +Δ로만 반영된다.

#### C. 전부 자체 (GTFS + OTP)

시간표를 자체 확보해 경로 탐색까지 직접 한다. **신호 대기를 비용함수 안에** 넣을 수 있어 "신호 때문에 더 나아지는 경로"를 후보로 만들 수 있다. → `stop_times` 수집만 **경기 약 2주**(§4.4)이고 그 위에 조립·구동이 얹힌다. 2주 예산엔 안 맞아 **보류**(폐기 아님).

### 2.2 왜 B+인가 — 이전 기각 사유의 재검토

| 이전 판의 C 선택 근거 | 실측 결과 |
|---|---|
| **#1. TMAP에 출발/도착 시각이 없다 → 어느 차를 탔는지 모름 → 대기 재계산 불가** | **절반만 맞다.** ✅ 시각 필드는 실제로 없다(`departureTime`/`arrivalTime` 0회). **그러나 `routeId`가 있다.** 시각표를 우리가 가지면 **남의 시각이 필요 없다** — 노선 순서만 받아 시각은 자체 전파한다. **기각 사유 소멸.** |
| #2. TMAP 대중교통 응답 도보에 `facilityType`이 없다 | ✅ 맞다 (`description`/`distance`/`linestring`/`streetName`뿐). **그러나 모든 WALK leg에 `start`/`end` 좌표가 있어** 보행자 API로 재조회하면 된다. 재조회는 선택이 아니라 **필수**. |
| #3. `searchDttm`은 출발 시각 하나만 | ✅ 맞다. 그래서 **TMAP의 시각을 아예 안 쓴다.** |
| #4. 자체 탐색이라야 신호를 비용함수에 넣을 수 있다 | ⚠️ **절반만 맞다.** 비용함수에는 못 넣지만 **TMAP의 `count` 최대 10개 후보를 우리 모델로 재정렬**할 수 있다. 못 잡는 건 "TMAP top-N 밖의 경로"뿐 → §2.3 |
| #5. C에선 TMAP 보행자 불필요 (OSM에 태그 있음) | ⚠️ §7.1 실측 — OSM 신호 커버리지 **~12%**. "TMAP 의존 0"은 성립하지 않는다. |

### 2.3 B+가 실제로 포기하는 것 — 생각보다 좁다

이전 판은 *"B는 남이 고른 경로를 **사후 채점**이라, 신호 때문에 더 나아지는 경로를 **후보에 올리지도 못한다**"*고 했다. **과하다.**

**TMAP은 `count`로 최대 10개 경로를 반환한다**(§3.6, 기본값 10). 그 10개를 **전부 우리 모델로 재계산해 재정렬**하면 경로 선택에 반영된다. "사후 채점"이 아니라 **후보군 내 재순위**다.

| | 반영되나 |
|---|---|
| TMAP이 준 N개 후보 중 **신호를 반영하면 순위가 뒤집히는 경우** | ✅ **재정렬로 잡힌다** |
| TMAP이 **애초에 후보에 올리지 않은 경로** | ❌ 이것만 못 잡는다 |

**C와의 실제 격차는 "TMAP의 top-N 밖에 있는 경로"뿐이다.** 그리고 신호 대기(수십 초~수 분)가 top-10 밖의 경로를 1위로 끌어올릴 만큼 크려면 상당한 차이가 필요하다 — **그 빈도는 측정된 적이 없다.**

**검증되지 않은 이득에 2주 예산 전체를 선불하는 것은 순서가 거꾸로다.** 2주로 만들어 재정렬이 실제로 얼마나 순위를 바꾸는지 보고, 그때 C를 재검토한다(§9 의사결정 지점).

> ⚠️ **`count`를 키우면 과금이 커질 수 있다** — §3.6이 지적하듯 itinerary당 과금인지 문서에 정의가 없다. 재정렬 이득과 비용을 같이 재야 한다.

> **C는 폐기가 아니라 보류.** §7.2에서 OTP 주입 리스크가 해소됐고 판교 클리핑이 2.6초라(아래) 되살리는 비용은 낮다. 되살릴 조건은 **§9의 의사결정 지점에서 "경로 선택에 반영"의 이득이 측정되는 것**이다.

### 2.3.1 ★ 확정 — 지역별 전략이 비대칭이다

**쿼터가 지역마다 자릿수로 달라서 같은 전략을 못 쓴다.**

| | **경기 (TAGO)** | **서울** |
|---|---|---|
| 쿼터 | **운영 50만/일** | **운영 1만/일** (1/50) |
| 규모 | 2,200 노선 · 146,362 구간 | 665 노선 · 38,534 구간 |
| **버스 전략** | **수집 + 조회** | **조회만** |
| 첫 정류장 절벽 | ✅ 도착정보 | ✅ 도착정보 (**더 풍부** — 위 표) |
| **지평 확장** (지평 밖 특정 차 지목) | ✅ **수집 → 자체 주행시간 모델** | ❌ **포기** |
| 중간 환승 버스 시각표 복원 | ✅ 수집(§9.1) | ❌ |
| CV | ✅ 실측 | ⚠️ `term`(계획 배차)만 |
| **지하철** | — | ✅ **파일(15098251) + `realtimePosition`** — 버스와 무관하게 됨 |

**서울은 지평 확장을 포기한다.** 쿼터상 선택지가 없고(152배 부족), **도착정보가 `nstnSec`·`term`·`goal` 을 직접 주므로 손실이 생각보다 작다.**

**경기는 수집한다** — B+ 동작 요건은 아니지만(§6.4.1) 버스에서 차별점을 보려면 필요하다.

### 2.4 구성 — 전부 확보됨 (신청·대기 없음)

| 필요 | 조달 | 상태 |
|---|---|---|
| 경로 뼈대 (`routeId`·역·도보구간 좌표) | TMAP 대중교통 | ✅ **구독됨** (§3.6) |
| 도보 실거리 + 횡단보도 위치 | TMAP 보행자 `facilityType=15` | ✅ **1,000/일** |
| 신호 주기·실시간 | 표준데이터 + 15157604 | ✅ bus-test에 구현됨 |
| **지하철 시각표 (절벽 판정)** | 서울교통공사 **파일** 15098251 | ✅ **쿼터 없음.** 1~9호선 405역, 경기 연장 포함 |
| **시각표 없는 노선** (신분당선 등) | 서울 열린데이터광장 `realtimePosition` 누적 | ✅ 키 발급됨 (§3.3.2 / §3.3.3) |
| 버스 기대 대기 | TAGO `intervaltime` | ✅ 이미 승인됨 |
| 버스 승차 시간 | TMAP `sectionTime` | ✅ **지하철 구간엔 쓰지 않는다** — 시각표가 이긴다 |

**→ B+ 는 `stop_times`(수집 2주)도, KTDB 도, OTP 도 필요 없다.** 수집은 C 를 되살릴 선택지를 위해 **백그라운드로 병행**한다(§4.4).

> 아래는 **기각된 C 안**의 구성이다. B+에는 OTP도 GTFS도 쓰지 않는다. §9의 의사결정 지점을 통과하면 여기서 재개한다.

```

### 3.1 버스 — TAGO (국토교통부, 서울 제외 전국)

**공통 베이스:** `https://apis.data.go.kr/1613000/`
**공통 파라미터:** `serviceKey`(URL Encode), `_type`(xml|json), `numOfRows`, `pageNo`
**공통 제약:** 개발계정 10,000건/일 · 운영 50만/일(✅ 확인)

> ### ⚠️ 정정 — "30 TPS · 평균 응답 500ms" 는 둘 다 틀렸다 (✅ 실측 2026-07-16)
>
> 활용가이드에 `초당 최대 트랙잭션 [30] tps` · `평균 응답 시간 [500] ms` 로 적혀 있으나 **실물과 다르다.**
>
> | 활용가이드 | 실측 |
> |---|---|
> | 평균 응답 **500ms** | **~3초** (6배) |
> | **30 TPS** 하나 | **제약이 두 개고 성격이 다르다** ↓ |
>
> | 실제 제약 | 증상 |
> |---|---|
> | **동시 세션 30개** | HTTP **200** + `resultCode 99` `가용한 세션이 존재하지 않습니다. (30/30)` |
> | **토큰 rate limit** | HTTP **429** `API token rate limit exceeded` |
>
> **① 세션 초과는 조용히 실패한다.** HTTP 200 이고 예외도 안 난다 → `resultCode` 를 안 보면 **"버스가 없다"로 읽힌다.** 여러 프로세스를 동시에 돌릴 때 **in-flight 합이 30을 넘지 않게** 할 것.
>
> **② rate 보다 버스트가 문제다.** 균등 6.0/s 는 429 가 0건인데, **같은 6.7/s 를 버스트로 던지면 78%가 터진다.** `ThreadPoolExecutor.map` 은 워커 수만큼 한꺼번에 던지므로 걸린다 → **제출을 `1/rate` 간격으로 벌릴 것.** 실측: 균등 2·4·6/s = 무결점, 10/s 부터 141/144.

> ### ⚠️ 정정 — 갱신주기는 공통 제약이 아니다 (✅ 활용가이드 원문 확인)
>
> 이전 판에 *"갱신 **일 1회**(포털의 '실시간' 표기는 오류, 활용가이드가 정확)"* 라고 적혀 있었다. **거꾸로다.**
>
> `오픈API활용가이드_국토교통부(TAGO)_버스위치정보v1.0.docx` 원문:
> ```
> 데이터 갱신주기      실시간(10~20초)
> 평균 응답 시간       [500] ms
> 초당 최대 트랙잭션   [30] tps
> ```
> **포털의 "실시간" 표기가 맞다.** 정적 데이터셋(노선·정류소)의 갱신주기를 공통란에 일반화하면서 위치·도착까지 휩쓸린 것으로 보인다.
>
> **영향:** 위치정보가 일 1회였다면 §4.4의 주행시간 수집이 성립하지 않아 **C가 그 자리에서 끝난다.** 실시간 10~20초이므로 수집 계획은 살아 있다. 또한 갱신이 10~20초이므로 **30초 폴링이 적정**하다 — 더 잦으면 같은 데이터로 쿼터만 태우고, 더 느리면 갱신을 건너뛴다.
>
> **30 TPS는 확인됨** — 상세기능마다 `초당 최대 트랙잭션 [30] tps`로 명시. 이 값이 §4.4 수집 규모를 지배한다.

#### ⚠️ 버스위치정보의 해상도는 **정류장 단위**다 (✅ 실측 2026-07-16 10:10)

311번(`GGB204000192`) 8대를 26~28초 간격으로 3번 찍은 결과:

```
10:10:23 → 10:10:51 (28초) : 0/8대 변화     ← 8대 전부 좌표조차 안 바뀜
10:10:51 → 10:11:17 (26초) : 2/8대 변화
   경기77바8433  순번 19→20  시흥사거리 → 성남농협영농종합지원센터
   경기70아8869  순번 68→69  모란고개 → 모란역5번출구
```

**28초 동안 8대 전부 좌표가 1mm도 안 움직였고, 바뀔 때는 정류장 단위로 점프한다.**
→ **`gpslati`/`gpslong`은 버스의 실제 GPS가 아니라 "현재 정류장의 좌표"다.** 정류장 사이에서 버스가 어디 있는지 이 API는 모른다.

**응답 필드 (✅ 실호출):** `vehicleno, routenm, routetp, nodeid, nodenm, nodeord, gpslati, gpslong`
⚠️ **타임스탬프 필드가 없다.** 서울 지하철 `realtimePosition`의 `recptnDt`(§3.3.2)와 달리 **데이터가 언제 측정됐는지 알 수 없다** → 우리 폴링 시각이 곧 관측 시각이고, 그만큼 오차가 얹힌다.

**이것이 카카오 사례를 설명한다:**

> 버스의 경우, **고정밀 위치 정보를 실시간으로 연동**하여 1초 단위의 위치 정보를 생성 … 기존 버스 도착 정보는 10~20초 단위로 갱신되어 **실제 주행 상황을 제대로 반영하지 못하는 한계**

**공개 API로는 1초 단위 궤적을 만들 수 없다.** 카카오가 직접 연동한 이유이고, 우리는 그 연동을 못 한다.

**그러나 우리 목적엔 이 해상도로 충분하다.** 필요한 것은 `stop_times` = **정류장 통과 시각**이지 정류장 사이의 위치가 아니다. 지도에 부드럽게 그릴 게 아니므로 카카오가 부족하다고 버린 해상도가 우리에겐 맞는다.

> **남는 제약:** 폴링 간격 = 통과 시각 오차. 30초 폴링이면 ±30초이고 정류장 간 1~2분 대비 25~50%다. 여러 날 누적하면 평균은 수렴하지만, **§4.4 로테이션 산수는 그대로 적용된다.**
> **갱신주기는 단정할 수 없다** — 28초에 0/8은 "그 사이 아무도 정류장을 안 넘었다"는 뜻일 수 있어 활용가이드의 `실시간(10~20초)`과 모순되지 않는다. **정류장을 넘어야 값이 바뀌기 때문.**

**주요 응답 필드 (✅ 311번 실호출)**

  startnodenm 사송동.종점 | endnodenm 위례31단지
  startvehicletime 0500 | endvehicletime 2240

```

#### ❌ 서울 버스 **수집**은 불가능하다 (조회는 별개 — §4.4)

```

운영계정 10,000/일 ÷ 2,280 = 4.4 노선          → 152배 부족
```

**운영전환이 이미 끝났는데 10,000 이다.** TAGO 운영이 50만인데 **서울은 1/50**이고 **더 올릴 경로가 없다.** 폴링을 30분으로 늘려도 263노선이라 665 에 못 미치고, 그 해상도는 무의미하다(정류장 간 1~2분).

> **서울시 버스 API 는 하루 1만 콜 규모로 설계돼 있다** — "정류장 몇 개 조회해 앱에 뿌리기"용이지 **전 노선 상시 폴링용이 아니다.** 우리가 하려는 게 그 설계 밖이다. **경기(TAGO)가 50만을 주는 게 예외적이다.**
>
> **→ 서울은 조회(요청당 1~2콜)로 B+ 에 붙이고, 수집은 경기만 한다.**

**그래도 남는 두 가지:**
1. **지평은 2대** — 모든 필드가 `1/2` 접미사, 즉 **1·2번째 도착예정뿐**이다. 그 버스들의 타임라인(주요정류소 3개 + 종점)은 풍부하나 **3번째 차는 없다**
2. **예측값이다** — `avgCf`(이동평균)·`expCf`(지수평활)·`kalCf`/`neuCf`(기타) 보정계수가 있는 걸 보면 **BIS 가 여러 모델로 예측**한 값이다. 실측이 아니다. 다만 **우리가 BIS 보다 잘 예측할 이유가 없으므로** 조회용으론 충분하다

#### ⚠️ 미해결 — `ws.bus.go.kr` 이 키를 거부한다

포털은 **`운영계정 | 연장신청` · `승인` · `2026-07-16 ~ 2028-07-16`** 인데 서버는 `Key인증실패: SERVICE KEY IS NOT REGISTERED ERROR.[인증모듈 에러코드(30)]` 를 반환한다. **위치·도착·노선 전부, 즉 서버 단위다.**

#### 3.3.3 위치정보 누적 — 용도가 바뀌었다

> ### 🔄 정정 — 시각표 복원은 필요 없어졌고, 애초에 순환이었다
>
> 이 절은 원래 *"신분당선 시각표가 없으니 `realtimePosition`을 누적해 복원한다"*였다. **두 가지가 틀렸다:**
>
> 1. **시각표는 이미 있다** — 운영사 공식 PDF (§3.3.1). 복원할 이유가 없다.
> 2. **더 나쁘게, 그 계획은 순환이었다.** §8 #1은 *"시각표 vs 실시간 → 정시성"*인데, **관측으로 복원한 시각표에 그 관측을 대보면 지연이 정의상 0이 나온다.** 신분당선의 정시성을 아예 못 재는 계획이었다.
>
> **→ 수집기의 용도는 "시각표 복원"이 아니라 "정시성 검증"이다.** 그리고 독립적인 계획 시각표(PDF)가 생겨 **지금 바로 잴 수 있다** — 며칠간 복원을 기다릴 필요가 없다.
>
> **복원이 여전히 유효한 경우:** 공식 시각표가 없는 노선(수인분당·경의중앙·경춘·경강·서해 등). 다만 그 노선들도 **먼저 운영사 공개자료를 찾을 것** — 신분당선이 그랬듯 data.go.kr에 없다고 없는 게 아니다.

**참고 — 복원이 지하철에서 성립하는 이유:** §4.4가 버스에 대해 이 방법을 기각했지만 **지하철에선 규모가 두 자릿수 다르다:**

| | 버스 | 지하철 |
|---|---|---|
| 대상 | **경기 2,200 노선** | **19개 노선** (신분당선만이면 **1개**) |
| 필요 콜 | 466만/일 | 45,600/일 (신분당선 2,400) |
| 잡음 | 도로·신호·정체 → 예측 오차 수 분 | **전용 궤도 → 정시성 높음** |
| 열차/차량 식별 | 불가 | **`trainNo`로 식별** — 매일 같은 번호가 같은 시각에 운행 |
| 수렴 | 수 개월~년 | **며칠** |

**`trainNo`가 결정적이다.** 버스는 배차간격 운영이라 "그 차"라는 게 없지만, 지하철은 열차번호가 매일 같은 시각표로 돈다 → **여러 날의 관측을 겹칠 수 있다.** 신분당선은 운행 열차가 20대뿐이라 며칠이면 수렴한다.

> ⚠️ 폴링 간격을 매일 **고정**하면 위상이 매일 같아 해상도가 안 올라간다. **매일 시작 오프셋을 흔들 것.**

> #### ✅ 실측 — 도보 leg 재조회는 **병렬로 한다** (2026-07-16, `spike/scripts/tmap_pedestrian_parallel.py`)
>
> 한 검색이 표시 경로 3개 × 도보 2~3구간 ≈ **~10콜의 보행자 재조회**를 요구한다.
> 순차면 ~2초(188ms × 10)라 UX 가 죽는다. 병렬로 던져도 되는지 실측:
>
> | 패턴 | 성공 | 지연 중앙값 | 벽시계 |
> |---|---|---|---|
> | 순차 3 (기준) | 3/3 | 188ms | 0.57s |
> | 동시 5 버스트 | 5/5 | 208ms | 0.23s |
> | **동시 10 버스트** | **10/10** | **243ms** | **0.36s** |
> | 동시 20 버스트 | 20/20 | 747ms | 0.80s |
>
> - **동시 20 버스트까지 전량 200. 429 도 차단도 없다** — TAGO(§3.1 버스트 민감)와 다르다.
> - **동시 10 이 스윗스팟.** 20 부터 개별 지연이 3배로 뛴다(차단이 아니라 서버 큐잉으로 보임).
>   검색당 재조회가 10 이하이므로 **검색 1회의 leg 들은 전부 동시에 던지면 된다** (세마포어 10 + 429 시 1회 재시도만 두고).
> - ⚠️ 이 실측은 **"한 검색의 버스트"까지만** 증명한다. 다수 사용자의 지속 TPS 상한은 별개 —
>   데모(순간 사용자 몇 명)엔 무관하고, 운영 전에 플랜 문서를 확인할 것.
>   어차피 **쿼터 1,000/일이 TPS 보다 먼저 바닥난다** (하루 ~100검색).

> #### ⚠️ 재정렬과 쿼터가 충돌한다 → **2단 재정렬** (§2.3 과 이 절을 잇는다)
>
> §2.3 은 *"TMAP 10개 후보를 **전부** 우리 모델로 재계산해 재정렬"*을 B+ 의 핵심 논거로 든다. 그런데 그걸 **문자 그대로** 하면 보행자 쿼터가 안 맞는다:
>
> | | 검색당 보행자 콜 | 하루(1,000/일) |
> |---|---|---|
> | 10후보 × 도보 3구간 전부 재조회 | **~30콜** | **33검색** ❌ |
> | 표시 3경로만 재조회 (위 병렬 실측 전제) | ~10콜 | ~100검색 |
> | **2단: 1차 근사 → 상위 2~3만 정밀** | **~6-9콜** | **110~160검색** ✅ |
>
> **2단 재정렬:**
> 1. **1차 (콜 0)** — TMAP 대중교통 응답이 이미 주는 **도보 `distance`** 로 신호 페널티를 **거칠게** 얹어 10개를 정렬한다. 횡단보도 수는 `distance` 에 비례한다고 근사 (재조회 없음).
> 2. **2차 (정밀)** — 1차 상위 **2~3개만** 보행자 API 로 재조회해 `facilityType=15` 횡단보도 위치·횡단거리로 정확히 계산, 최종 순위 확정.
>
> **못 잡는 것:** 1차에서 상위 3위 밖으로 밀렸는데 정밀 계산하면 1위가 될 경로. 하지만 신호 페널티가 그만큼 순위를 뒤집으려면 도보 거리 차이가 커야 하고, 그건 1차 근사가 이미 대략 잡는다. **§2.3 이 "top-N 밖 경로만 못 잡는다"고 한 것과 같은 성격의 잔여 손실**이고, 쿼터를 3배 아끼는 값이 그보다 크다. B+ 구현(§9 #4) 은 이 2단 방식으로 한다.

- 약관 ([대중교통](https://transit.tmapmobility.com/terms) · [TMAP API](https://tmapapi.tmapmobility.com/terms.html) — `/terms`는 404, **`.html` 필수**):
  > TMAP Open API를 이용하여 얻어진 데이터는 **저장 후 24시간 이상 사용할 수 없습니다.**
  > 동일한 서비스를 위하여 **다수의 프로젝트를 생성**하여 사용하는 경우에는 **불법 사용으로 간주**
  - 카카오식 전면 금지가 아니라 **TTL 24시간 상한**. **저장 시점 기준**이므로 sliding 금지 (`expireAfterWrite` O, `expireAfterAccess` ❌)
- **비용 감** (유저 100건/일 × 1만 DAU = 100만콜/일): 월 약 2,640만원 / 연 3.2억원. 유저당 2,640원/월. → C로 가면 **0**

### 3.7 data.go.kr 키 운영 (자주 헷갈림)

- 활용신청은 **API별**이며, 승인되면 **그 키의 권한 목록이 늘어난다.** 403이면 키를 다시 받는 게 아니라 해당 API에 신청할 것

- **✅ 운영단계도 자동승인이다** (심사 아님). 포털 원문: *"개발계정 : 10,000 / 운영계정 : **활용사례 등록시 신청하면 트래픽 증가 가능**"* → 동작하는 서비스를 활용사례로 등록하면 됨. **bus-test가 그 요건을 이미 충족**하므로 닭-달걀 문제는 없다.
  - ⚠️ **증가 한도는 어디에도 명시돼 있지 않다.** 이 값이 §4.4 일정을 결정한다 → §8 #2
- **✅ 에러 문자열로 원인 구분 (실측 확인):** 미승인 API → `Forbidden`(10 bytes) / 등록 안 된 키 → `Unauthorized`(12 bytes)

### 4.4 ❌ 결정적 결측 — 정류장 간 주행시간

`getRouteAcctoThrghSttnList` 응답 **전체**:
```
routeid, nodeid, nodenm, nodeno, nodeord, gpslati, gpslong, updowncd
```
활용가이드 전체에 **`도착시각` 0회 · `소요시간` 0회 · `통과` 0회 · `arrtime` 0회.**

**GTFS `stop_times.txt`는 정류장마다 `arrival_time`/`departure_time`이 필수인데, 그 값이 어느 API에도 없다.**

| GTFS 필요 | TAGO 제공 |
|---|---|
| `stop_id`, `stop_sequence` | ✅ `nodeid`, `nodeord` |
| 정류장 좌표 | ✅ `gpslati`/`gpslong` |
| `direction_id` | ✅ `updowncd` |
| 배차간격 | ✅ `intervaltime` 등 |
| **`arrival_time`/`departure_time`** | ❌ **없음** |

**채우는 방법**

| 방법 | 즉시성 | 정확도 |
|---|---|---|
| 좌표 거리 ÷ 가정 속도 | 즉시 | ❌ 조악 — **KTDB+OTP의 10분 오차가 이 계열** |

> ### 🔄 정정 — 이전 판의 "7개월"은 지어낸 숫자였다
>

> 결론(2주 예산엔 안 맞으므로 B+ 채택)은 유지되지만 **이유가 다르다.** §9 "C 재검토"는 7개월 뒤가 아니라 **2주 뒤**의 얘기다.

**입력 (✅ 실측)**

| 값 | 근거 |
|---|---|
| 경기 **2,200 노선 · 146,362 구간** | `getRouteNoList` + `getRouteAcctoThrghSttnList` 31개 시군 전수. 노선당 66.5구간 |

**⚠️ 제약은 "30 TPS"가 아니라 두 개다 (✅ 실측)**

| 실제 제약 | 증상 |
|---|---|
| **동시 세션 30개** | HTTP **200** + `resultCode 99` `가용한 세션이 존재하지 않습니다. (30/30)` |
| **토큰 rate limit** | HTTP **429** `API token rate limit exceeded` |

**그리고 진짜 변수는 rate 가 아니라 버스트다:**

```
균등  6.0/s (167ms 간격 제출)  →  88건 전부 성공, 429 0건
버스트 6.7/s (20개 동시 투입)  →  180건 중 140건 실패 (78%)
```
같은 평균인데 결과가 정반대다. `ThreadPoolExecutor.map` 은 워커 수만큼 한꺼번에 던지므로 토큰 버킷에 걸린다. **제출을 `1/rate` 간격으로 벌리면 해결된다** (실측: 균등 2·4·6/s = 429 0건, 10/s 부터 141/144로 섞이기 시작).

> ⚠️ **세션 초과는 조용히 실패한다.** HTTP 200 이고 예외도 안 나므로 `resultCode` 를 안 보면 **"버스가 없다"로 읽힌다.** 실제로 그 함정에 빠져 요청의 2/3를 모르게 버렸다. **`resultCode` 를 반드시 검사하고 실패는 항상 로그에 찍을 것.**

**소요 (균등 6/s 기준)**

```
6/s × 30초        = 180 노선/사이클
6/s × 68,400초    = 410,400 콜/일          ← 쿼터 47만의 87%
필요 관측 = 146,362구간 × 7밴드 × 7샘플     = 7,171,738
관측률   = 180노선 × 3,823/일              =   688,140/일
→ 10.4 평일 ≈ 2주
```

| 개발 1만/일 | 4.4 | 468 라운드 → 비현실적 |
| **운영 50만/일 (✅ 확인)** | **180** | **약 10 평일 = 2주** |

> ### ⚠️ 분석 규칙 (2026-07-18) — 요일 셀은 사실상 '단일 날짜' 표본이다

> **수집으로 풀지 않는다** — target 을 지하철처럼 '관측 일수'로 세면 요일당 7주 = 총 수십 주라 기각했다(config `_dayTypes`). **분석 단계 규칙으로 다룬다:**
>
> | # | 규칙 | 근거 |
> |---|---|---|
> | 1 | **요일 간 비교**(요일 효과 판정 · 화수목 재통합 판단)에는 **같은 요일의 서로 다른 날짜 ≥ 2 인 셀만** 쓴다. 날짜 수는 장부 `n`(건수만 안다)이 아니라 **jsonl 의 `t` 날짜**로 센다 | 날짜 1개면 요일↔날짜 판별 불가. 2개부터 날짜 간 분산이 보인다 |
> | 2 | 날짜 1개뿐인 셀도 **밴드별 주행시간 모델(`stop_times`)에는 쓴다** | 편향이 아니라 분산의 문제라 평균 추정엔 유효하다. 로테이션 2회전부터 날짜가 자연히 쌓인다 |
> | 3 | 요일별 차이를 보고할 땐 **날짜 간 분산을 함께** 제시한다 — 셀 내 7표본의 분산이 아니라 | 셀 내 분산은 폴링 오차만 반영해 실제 불확실성을 과소평가한다 |
>
> 2회전(~2주 뒤 §9 의사결정 지점의 층화 표본 검증)부터는 같은 요일이 2날짜 이상 자연 축적되므로 규칙 1 은 시간이 해결한다 — **초기 데이터로 요일 판정만 내리지 않으면 된다.**

**⚠️ 서울은 별개다** — TAGO 에 없다(§4.1). 위치정보는 **`15000332`**(`ws.bus.go.kr/api/rest/buspos`, ✅ 확인). 개발 **1,000/일**이라 375노선 × 2,280사이클 = **855,000/일**의 855배 부족 → **운영전환 필수이고 그 한도는 미확인.** 위 "경기+서울" 표는 **서울 쿼터가 경기와 같다는 미검증 가정** 위에 있다.

#### 그럼에도 B+ 를 택하는 이유는 남는다

2주는 **구현 예산 전체**이므로 수집이 끝나야 시작할 수 있는 C 는 여전히 안 맞는다. 그러나 **부채의 근원은 "경로 탐색을 자체로 한다"는 결정**이다. 탐색을 TMAP 에 맡기면 버스 승차 시간을 남이 주므로 **`stop_times` 가 통째로 불필요해진다.**

**TMAP 의 `sectionTime` 이 우리가 2주 모아 만들 값보다 나쁠 이유도 없다** — TMAP 은 실시간 교통정보를 가진 사업자다. 위 표의 "바닥 오차 10분"은 **좌표거리÷가정속도**로 할 때의 값이지 TMAP 값을 쓸 때가 아니다.

> **→ 수집은 계속한다.** C 를 되살릴 선택지를 위해 백그라운드로 돌린다. 2주 뒤 §9 의사결정 지점에서 C 를 재검토할 때 **데이터가 이미 있다.**

**⚠️ 지하철에는 이 산수가 적용되지 않는다.** 노선이 19개뿐이고 `trainNo` 로 열차를 식별할 수 있어 며칠이면 수렴한다 → §3.3.3

**해상도를 낮춰 맞추는 건 답이 아니다:** 2,200노선을 30 TPS에 넣으면 73초 간격인데, 도심 정류장 간격이 1~2분이라 **오차가 재려는 값과 같은 크기**가 된다. 범위를 줄이면(900노선=경기의 41%) GTFS가 나머지 59%를 모른 채 경로를 고르므로 **부분 커버가 아니라 틀린 답**이 나온다 — §4.1의 "미지원 지역을 명시하면 해소된다"는 논리가 여기선 안 통한다. 지역 단위 구멍이 아니라 **같은 지역 안에서 노선이 반쯤 빠지는** 것이라서.

> 참고: 카카오의 '버스 예상시간표'가 **"과거 운행 기록을 통해 정류장 도착 시간을 예측"**하는 것이 이것. 요일별·시간별 + 최빠름/최늦음 범위까지 제공 = **분포를 보유**. 실시간을 대체한 게 아니라 **BIS가 침묵하는 구간(예보 지평 밖)을 메우는 보완**이다. 수년치 축적이 해자.

**`frequencies.txt`가 핵심 이점:** 시내버스는 시각표가 존재하지 않고 **배차간격으로 운영**된다. GTFS 표준이 이를 정식으로 표현하므로 **시각표를 지어낼 필요가 없고**, 배차마다 trip을 만들지 않아 `stop_times`가 템플릿 크기로 유지된다(전국이라도 수백 MB 수준, CSV로 충분).

## 6. 대기 계산 모델

### 6.1 절대 시각 앵커링

실시간 도착 API는 **조회 시점** 기준 "N분 남음"이다. 반드시 절대 시각으로 고정할 것.

```
T_i = t0 + arrivalsSec[i]                    t0 = 조회 시각
t1  = 출발 + 보정 도보 + 신호 대기            = 정류장 도착 시각
대기 = min{ T_i − t1 | T_i ≥ t1 }
```

### 6.2 ⚠️ 예보 지평

BIS 실시간은 **다음 2대까지만** 준다 (`predictTime1`, `predictTime2`). 실서비스의 첫 도보는 5~15분이라 **두 대 다 지나가고 3번째는 추적 범위 밖 → API가 침묵**한다. 게다가 멀수록 예측 드리프트도 크다. **두 문제가 같은 방향 → 도보 시간이 이 방식의 유효 경계.**

**→ 실시간은 탐색 시점이 아니라 "정류장 접근 중 / 도착 후"에 쓰는 것.**

#### 정정 — 지평은 배차에 따라 다르고, 스스로 상쇄된다

"두 대 다 지나간다"는 **도심 촘촘한 배차** 얘기다. **2대 지평 = 배차 × 2** 이므로:

| 배차 | 2대 지평 | 도보 10분 커버? | 절벽 크기 |
|---|---|---|---|
| **20분** (성긴 노선) | **40분** | ✅ | **크다** |
| 10분 | 20분 | ✅ | 중간 |
| 3분 (도심 촘촘) | 6분 | ❌ | **작다** |

**절벽이 큰 노선(성긴 배차)일수록 지평이 넓다.** 지평이 좁은 노선은 애초에 절벽이 작아 상관없다. → **지평 문제가 스스로 상쇄된다.**

⚠️ **단 "지금 조회한 값"이라는 건 그대로다.** 반드시 §6.1 대로 **조회 즉시 절대 시각으로 앵커링**해야 도보 10분 뒤와 비교할 수 있다.

#### ⚠️ 서울 `getArrInfoByRouteAll` — 노선 전 정류소를 1콜에

`15000314` 의 `getArrInfoByRouteAllList` 는 **경유노선 전체 정류소의 도착예정**을 한 응답으로 준다 (각 정류소마다 다음 2대 + 배차간격 + 다음 정류소 소요시간). 정류소별로 따로 부를 필요가 없다.
⚠️ 그래도 **노선당 1콜**이므로 수집 쿼터 산수는 위치정보와 같다 (§3.2).

### 6.3 절벽(cliff)은 시각표가 있는 곳에만 존재한다

> ### ⚠️ 이전 판의 기준(`H`)은 틀렸다
>
> 이전 표는 `H>20분`이면 절벽, `H<10분`이면 없음으로 갈랐다. **그런데 §6.4는 같은 문서에서 311번(H=20분)을 배차간격으로 다뤄 "절벽을 만들지 않고 단순 +90초"라고 한다. 6.3과 6.4가 서로 어긋난다.**
>
> **`H`는 축이 아니다. 축은 ① 시각표의 존재 ② 그것을 실제로 지키는가 두 개다.**

| 수단 | 시각표 | 지키는가 | 절벽 |
|---|---|---|---|
| **지하철** | ✅ | **⚠️ 미검증** — 전용 궤도라 유력하나 ✅ 없음 → §8 #2 | ✅ **차별점 최대** |
| **광역·직행좌석 버스, 기점 근처** | ✅ 유력 | ✅ 출발시각 확정 | ✅ **가능** |
| 광역 버스, 중간 정류장 | ✅ 유력 | ❌ 도로에서 밀림 | ⚠️ **흐림** |
| **도심 시내버스** | ❌ 배차간격 운영 | — | ❌ **없음 — 그냥 +Δ** |
| 실시간 지평 안 (정류장 근접) | — | BIS 실시간 | ✅ 절벽 (그 차를 놓침) |

배차간격 운영이면 정류장 도착이 배차 주기에 대해 사실상 무작위 → **기대 대기가 도착 시각과 무관** → Δ가 붙어도 총 시간은 정확히 +Δ. 같은 버스라도 **탐색 시점엔 통계, 정류장 근접 시엔 절벽**으로 성격이 바뀐다.

#### ⚠️ 역설: 불규칙성은 절벽을 만드는 게 아니라 없앤다

"실제로는 배차대로 안 다니고 교통 상황에 따라 달라지지 않나"는 **절벽을 강화하지 않는다.** 절벽이 성립하려면 **그 차가 언제 오는지 알아야** 한다 — 특정 차를 "놓친다"는 게 성립하려면 도착 시각이 예측 가능해야 한다. 교통에 따라 흔들리면 지목할 대상 자체가 없다. §6.4 공식으로 보면 CV가 커질수록 `E[대기]`는 **커지기만** 하고 도착 시각과의 상관은 여전히 0이다. **불규칙할수록 90초는 더 안 보인다.**

**그리고 버스는 우리가 모델링하려는 바로 그 신호에 자기도 걸린다.** 10분 상류의 버스는 신호 교차로를 10~15개 통과하며 도착 지터가 수 분이 된다. **재려는 값(90초)보다 잡음이 크다.** 지하철이 다른 이유가 정확히 이것 — 전용 궤도라 신호에 안 걸린다.

> **버스위치정보로 지평을 넓혀도 마찬가지다.** 위치정보는 도착정보의 "다음 2대" 한계를 넘게 해주지만(§6.2), **위치를 열어주는 구간이 정확히 예측이 무너지는 구간**이다. 가까우면 도착정보로 충분하고, 멀면 위치를 알아도 90초를 분간 못 한다. **순증분이 없다.**
> → 버스위치정보의 제 용도는 실시간 판정이 아니라 **오프라인 주행시간 통계**다(§4.4·§5). 단 §4.4 결론대로 그 경로는 7개월이라 **B+에선 쓰지 않는다.**

#### ⚠️ 버스 대기의 불확실성이 신호 90초를 삼킨다

§6.4의 `E[대기] = (H/2)(1+CV²)`에서 **CV를 아무도 주지 않는다.** TAGO `intervaltime`은 **계획** 배차지 실제 뭉침이 아니다.

| CV 가정 | 20분 배차의 기대 대기 |
|---|---|
| CV=0 | **10분** |
| CV=1 | **20분** |

**모르면 10분 차이인데 재려는 신호는 90초다.** §1의 논리("신호가 바닥 오차보다 커야 측정된다")가 버스 대기에서 그대로 재현된다. **그리고 이것은 아키텍처 문제가 아니다 — C로 가도 같다.** `stop_times`는 정류장 간 **주행시간**이지 배차 **CV**가 아니다. 7개월을 태워도 이 10분은 없어지지 않는다.

**→ 버스는 "TMAP `sectionTime` + 기대 대기"로 두고 더 손대지 않는다.** 신호 Δ는 정확히 반영되지만(절벽이 없으므로 근사가 아니라 정확) 그 정확함이 눈에 띄지 않는다.

#### 신호 × 절벽의 실제 영토

| 구간 | 신호 | 절벽 | TMAP으로 불가 |
|---|---|---|---|
| 첫 도보 → 배차 버스 | ✅ | ❌ | — |
| 첫 도보 → 지하철 | ✅ | ✅ | ❌ 출발 시각이라 보정 가능 |
| 지하철 → 도보 → 지하철 | ❌ **역 안엔 횡단보도가 없다** | ✅ | ✅ |
| **버스/도보 → 도보 → 지하철** | ✅ | ✅ | ✅ |

**마지막 행이 차별점이 성립하는 유일한 조합이다.** 나머지는 신호가 없거나(역 안), 절벽이 없거나(배차 버스), TMAP으로도 되거나(첫 도보) 셋 중 하나다. §3.6 실측에서 TMAP이 반환한 첫 경로가 정확히 이 형태(`버스 → 환승도보 → 신분당선`)였다.

### 6.4 대기시간 역설 (흔한 실수)

배차 H면 기대 대기가 H/2가 **아니다.**

```
E[대기] = (H/2) × (1 + CV²)        CV = 배차간격의 변동계수
```

버스 뭉침(bunching)으로 CV가 크고, **CV=1이면 기대 대기 = H**. H/2로 잡으면 대기를 체계적으로 과소평가한다.

> 실측 예: 311번 평일 배차 20분 → 기대 대기 최소 10분. 신호 90초는 이 노선에서 절벽을 만들지 않고 단순 +90초.

### 6.4.1 ★ B+ 의 버스 처리 — 세 분기 (결론)

**도착정보 하나로 다 되지 않는다.** `도착정보는 조회 시점 기준 "N초 후"` 이므로 §6.1 앵커링으로 절대 시각을 박아야 하고, 그것도 **첫 정류장에서만** 유효하다.

| 구간 | 근거 | 절벽 | 신호 90초의 효과 |
|---|---|---|---|
| **첫 정류장 + 지평 안** | 도착정보 앵커링 (§6.1) | ✅ | **배차만큼 증폭** |
| 첫 정류장 + 지평 밖 | 기대 대기 `(H/2)(1+CV²)` | ❌ | 그냥 +90초 |
| **중간 환승 버스** | 기대 대기 | ❌ | 그냥 +90초 |
| *(대조)* 지하철 전 구간 | **자체 시각표** | ✅ | 열차 간격만큼 증폭 |

**중간 환승 버스가 안 되는 이유 — 두 겹이다:**
1. **도착정보는 "지금"만 안다.** 30~40분 뒤 환승 정류장에 올 버스는 **지금 차고에 있다.** 조회해도 없다
2. **"그때 다시 조회"도 안 된다.** 사용자는 **지금** 경로를 묻고 우리는 **지금** 총시간을 답해야 한다

→ **§6.3 "신호 × 절벽의 실제 영토"에서 `버스/도보 → 도보 → 지하철` 만 남는 이유가 이것이다.**

#### ⚠️ 그래서 수집이 되살아난다 — "동작 요건"은 아니지만 "정확도의 대부분"

**B+ 는 수집 없이 돈다.** 도착정보 조회만으로 첫 정류장 절벽이 잡힌다. **그러나 위 표의 2·3행이 전부 `+90초`로 뭉개지면 §1 의 차별점이 지하철에서만 보인다.**

| 개선 대상 | 필요한 것 | 출처 |
|---|---|---|
| 첫 정류장 **지평 밖** | **위치정보 + 주행시간 모델** → 특정 차 지목 | D0 수집 |
| **중간 환승 버스** | **이력으로 시각표 복원** (성긴 배차 노선만 — §9.1) | D0 수집 |
| 기대 대기 정확도 | **CV 실측** — 모르면 10분 오차(§6.4) | D0 수집 |
| TMAP `sectionTime` 검증 | 대조군 | D0 수집 |

> **판단 이력 (같은 질문을 세 번 뒤집었다):**
> ① "수집 필요" → ② "B+ 엔 불필요, 조회로 됨" → ③ **"동작엔 불필요, 버스 차별점엔 필요"**
> ③ 이 결론이다. **수집 없이도 돌지만, 버스에서 차별점을 보려면 수집이 필요하다.**

### 6.5 데이터 요구의 역설

**구하기 어려운 쪽(시내버스 시각표)은 필요 없고**(배차간격만 있으면 됨 — TAGO가 요일별 제공), **필요한 쪽(지하철)은 구하기 쉽다**(운영기관 6~7개, 각자 최신 시각표를 요일 구분해서 공개).

### 6.6 안전 마진

`t1`이 `T_1`보다 20초 이르면 산술상 탑승이나 실제로는 놓칠 수 있다. **30초~1분 마진**을 두고, 마진 안이면 "빠듯함" 표시 또는 다음 차 기준. 신호 하나에 버스가 갈리는 케이스라 오히려 제품 취지에 부합하는 표시.

## 7. 도보 + 신호 모델

### 7.1 신호 대기를 OSM 도보 엣지에 반영

**핵심 통찰: 실시간 신호 위상은 유효기간이 매우 짧다.** 지금 초록인 건 알 수 있어도, 30분 뒤 환승 지점의 위상은 감응식·TOD로 주기가 바뀌어 외삽이 무너진다.

| 구간 | 근거 | 성질 |
|---|---|---|
| **뒤쪽 도보 (환승 등)** | 기대 대기 = `red² / (2×cycle)` | **상수** → OSM 엣지 가중치에 미리 반영. 알고리즘 수정 0 |
| **첫 도보 구간만** | 실시간 위상 | 경로 산출 후 시각만 후처리 보정 |

전처리 배치: **전국횡단보도표준데이터 ↔ OSM `highway=crossing`/`crossing=traffic_signals` 좌표 매칭** → 각 엣지에 기대 대기 반영. 오프라인이라 쿼리 지연에 영향 없고, 좌표 근접 매칭에 공을 들일 여유가 있다.

#### OSM 신호 커버리지 (✅ 실측 — 판교 bbox 127.06~127.13 / 37.38~37.43)

| 태그 | 노드 | 웨이 |
|---|---|---|
| `highway=footway` (보도 전체) | — | 7,629 |
| `footway=sidewalk` | — | 3,559 |
| **`footway=crossing`** (횡단보도) | — | **1,518** |
| `highway=crossing` | 305 | — |
| ├ `crossing=traffic_signals` (구 스킴) | **88** | **187** |
| ├ `crossing:signals=yes` (신 스킴) | **58** | **0** |
| └ `crossing=uncontrolled,marked` (무신호) | 89 | — |
| `highway=traffic_signals` | 106 | — |

**좋은 소식:** 판교 OSM은 **인도 단위로 그려져 있다**(sidewalk 3,559). §2의 "OSM에 있음 → TMAP 의존 0"이라는 전제 자체는 살아 있다.

**나쁜 소식 셋:**

1. **두 스킴이 겹치지 않는다.** 88 + 58 = 146인데 **합집합이 144** → 중복 2개뿐, 사실상 별개 노드다. `crossing=traffic_signals`만 필터하면 **58개(40%)를 조용히 잃는다.** 타임스탬프상 `crossing:signals=yes`는 **2026-02, 2026-03에도 편집 중** — 진행 중인 리태깅이라 이 비율은 계속 움직인다.
2. **커버리지 ~12%.** 신호 정보가 있는 건 144노드 + 187웨이인데 횡단보도 웨이는 1,518개. **나머지 88%는 신호 유무 자체가 미상.**
3. → **표준데이터 매칭은 보조가 아니라 주력이어야 한다.** OSM만 믿으면 8곳 중 7곳이 빈칸이다. 이 문서가 매칭을 "전처리 배치" 한 줄로 처리했지만, 실제로는 **여기가 데이터 파이프라인의 본체**다.

### 7.2 ✅ 해소됨 — OTP는 이미 보행 신호 대기를 넣고 있다

이전 판은 이곳을 *"유일한 미검증 리스크 / 표준 OSM 태그가 없다 / 막히면 C 전체가 막힌다"*로 두고 **거리 뻥튀기 vs fork**를 저울질했다. **전제가 틀렸다.** OTP 2.9 소스 확인 결과:

```java
// street/.../intersection_model/SimpleIntersectionTraversalCalculator.java:189
private double computeWalkingTraversalDuration(IntersectionVertex v, ...) {
  final var baseDuration = computeNonDrivingTraversalDuration(from, to, toSpeed);
  return v.hasWalkingTrafficLight()
    ? getExpectedWalkingAndCyclingTrafficLightTimeSec() + baseDuration   // = 15초
    : baseDuration;
}
```

**기본값(`intersectionTraversalModel: simple`)으로 신호 횡단보도마다 15초를 도보 통과시간에 더한다.** 이미 라우팅 비용에 반영돼 있다.

→ **거리 뻥튀기도 fork도 불필요.** 지오메트리를 왜곡할 이유가 없다. 그리고 주입 지점이 깔끔하게 열려 있다:

| 요소 | 실물 |
|---|---|
| 인터페이스 | `IntersectionTraversalCalculator.computeTraversalDuration(v, from, to, mode, ...)` — **`IntersectionVertex v`를 받는다** → 정점 좌표로 표준데이터 조회 가능 |
| 선택 | `enum IntersectionTraversalModel { SIMPLE, CONSTANT }` + 팩토리 `create()` switch |
| config | `/routingDefaults/intersectionTraversalModel` = `simple`\|`constant` (RouteRequest, since 2.2) |

→ **커스텀 계산기 = 클래스 1개 + enum 값 1개.** 이전 판이 각오한 "OTP 내부 파악 + 업스트림 추적 부담"과는 규모가 다르다.

#### 대신 새로 드러난 문제 셋

**① OTP는 신 태그 스킴을 모른다 (✅ 소스 확인)**

```java
// IntersectionVertex.java:46
public boolean hasWalkingTrafficLight() {
  return BitSetUtils.get(flags, CROSSING_TRAFFIC_LIGHT_INDEX);   // ← crossing=traffic_signals 단독
}
// OsmNode.java:33 ← VertexGenerator.java:168 ← crossing=traffic_signals 만 파싱
```
`crossing:signals`는 **OTP 소스 전체에 0건.** §7.1 실측과 겹치면 **판교에서 88개는 15초, 신호가 있는 58개(40%)는 0초**로 처리된다. 신 스킴이 계속 느는 중이라 방치하면 손해가 커진다. → `OsmNode` 패치 필요.

**② 중복 계상은 OTP가 자인한 알려진 버그 (원문 주석)**

> *"Sometimes this is applied twice (or even more than twice) when crossing a road which has multiple traffic lights which are synchronized. Therefore, this penalty should not be too high, so it's reasonable on average until we have implemented some logic for combining these traffic lights in intersections."*

**상수를 15초로 낮게 잡아 버그를 뭉갠 것이다.** 우리가 최악값 60초를 넣으면 도로 하나 건너는 데 **120초 이상**이 붙는다 → §7.4

**③ 베이스라인이 0이 아니다**

§1의 가설은 *"기존 길찾기는 도보 구간에 신호 대기가 0초"*지만 **OTP는 이미 15초를 넣는다.** §9의 "신호 반영 vs 미반영 차이 측정"은 **0이 아니라 OTP 기본 15초와 비교**해야 한다. 이걸 놓치면 차이를 부풀려 읽는다.

#### 게이트 실험이 공짜가 됐다

`intersectionTraversalModel`이 RouteRequest config이므로, "신호 비용이 경로를 바꾸는가"는 **코드 없이 `simple` ↔ `constant` 플립만으로** 볼 수 있다. 갈리면 게이트는 이미 통과이고, 남는 질문은 **"OTP의 뭉뚱그린 15초 대신 횡단보도별 실제 값이 추가로 경로를 바꾸는가"**로 좁혀진다 — 차별점이 걸린 곳은 거기다.

### 7.3 신호 실시간의 요청 경로 배제

- `crsrd_map_info`는 **정적**(교차로 좌표 4,239건) → **부팅 시 1회 적재 + 공간 인덱스**. 요청마다 조회 금지 (전국 목록을 순차 페이징하면 ~3초 소모)
- `tl_drct_info`는 `crsrdId` 서버 필터가 안 되고 **전국 목록을 통째로** 준다 → 횡단보도가 몇 개든 **응답 하나로 커버**. 백그라운드 1~2초 폴링으로 메모리 스냅샷 → 요청 경로에서 **0콜 / 0ms**
- 공공데이터라 TMAP의 24시간 제약과 무관

### 7.4 ⚠️ 대기 모델을 하나로 정해야 한다 (미결)

현재 세 곳이 서로 다른 모델을 쓴다.

| 출처 | 모델 |
|---|---|
| 이 문서 §7.1 | 첫 도보 = 실시간 위상, 뒤쪽 = 기대값 `red²/(2×cycle)` |
| bus-test | **최악값**(적색 전체) — "의도적, 기댓값 아님" |
| UI 데모(`route_finder_map.tsx`) | 모든 횡단보도의 `(t+offset) % cycle` 결정론적 위상 |

**UI 데모의 모델은 성립하지 않는다.** 실시간 API에 **주기·오프셋 필드가 없고**(§3.4), §7.1이 말하듯 30분 뒤 위상은 외삽이 무너진다. 걷는 속도로 초록 걸림이 달라지는 연출은 **첫 횡단보도에서만** 가능하다.

**엣지 가중치에는 실시간 위상을 넣을 수 없다** — 그래프는 빌드 시 1회, 위상은 매초 바뀐다. 따라서 "가능하면 실시간, 아니면 최악값" 규칙은 **비용에 대해선 항상 최악값 분기**로 떨어진다. 실시간 위상은 경로 산출 후 **첫 도보 구간 표시를 보정**하는 데만 쓰인다.

⚠️ **최악값은 기대값의 3배다** (red=60·cycle=90 → 최악 60초 vs 기대 `60²/(2×90)`=20초). 여기에 §7.2 ②의 중복 계상이 겹치면:
- **§9 1단계 게이트가 과장된 페널티로 쉽게 통과한다** — 경로는 바뀌지만 인공물이다
- **§6.3의 절벽과 만나면** 탈 수 있는 차를 "놓친다"고 판정해 총시간이 배차간격만큼 통째로 튄다. §6.6의 안전 마진은 이걸 다듬으라는 것이지 최악값으로 밀라는 것이 아니다

→ **게이트 실험은 최악값(60s)과 기대값(20s) 양쪽으로 돌려 비교할 것.** 기대값에서도 경로가 바뀌면 그때가 진짜다.

#### ★ 통과 가능 여부 — 걷는 속도가 신호를 가른다 (지금 어디에도 없는 계산)

현재 모델도 UI 데모도 **"녹색이면 통과"**다:

```js
const wait = phase < cw.green ? 0 : (cw.cycle - phase);   // 녹색이면 대기 0
```

**틀렸다.** 녹색이 1초 남았을 때 도착하면 대기 0으로 치는데, **1초에 16m 를 못 건넌다.**

실제 조건:

```
녹색 잔여   = green − phase
내 횡단시간 = 횡단거리 ÷ 내 속도
통과        ⟺ 녹색 잔여 ≥ 내 횡단시간
→ 유효 녹색 창 = green − 횡단거리÷내속도      (green 이 아니다)
```

**속도가 유효 창을 2배로 벌린다** (16m · 녹색 23초 기준):

| 걷는 속도 | 횡단 소요 | **유효 녹색 창** |
|---|---|---|
| 3.6 km/h (1.0 m/s) | 16초 | **7초** (23초의 30%) |
| 5.0 km/h (1.4 m/s) | 11.4초 | **11.6초** (50%) |
| 7.2 km/h (2.0 m/s) | 8초 | **15초** (65%) |

**이것이 차별점의 핵심 메커니즘이다.** §1 가설은 *"걷는 속도가 총시간을 바꾼다"*인데, 지금 모델에선 속도가 **도보 시간에만 선형으로** 영향을 준다. 이 조건을 넣으면 **속도가 신호 통과 여부를 바꿔 주기 하나(70~160초)를 통째로 점프**시킨다 — **지하철 절벽과 같은 구조**이고, 신호에서 절벽이 나오는 유일한 자리다.

> ⚠️ **세 가지 속도를 혼동하지 말 것:**
>
> | 속도 | 쓰이는 곳 | 사용자 설정을 따르나 |
> |---|---|---|
> | **설계속도 1.0 m/s** | 녹색 시간 **산정** (경찰청 매뉴얼) | ❌ **신호기 속성** — 빨리 걷는다고 녹색이 짧아지지 않는다 |
> | 사용자 속도 | 도보 이동 + **횡단 소요** | ✅ |
> | 사용자 속도 | **통과 판정** (위 조건) | ✅ |
>
> ⚠️ **점멸 구간은 더 좁다.** 법규상 녹색점멸 중 진입 금지이므로 유효 창이 더 줄어든다. 실제 보행자는 진입하므로 모델링 여부는 판단 필요.

### ✅ 해소된 것

| 이전 판의 과제 | 결과 |

| 신분당선 데이터 부재 | ✅ **두 번 오판했다.** ① 실시간: `realtimePosition`이 판교 포함 전 구간 커버(§3.3.2) ② **시각표: 운영사 공식 PDF가 있다**(§3.3.1) — data.go.kr에 없다고 "없다"고 결론낸 것 |

> **#1이 지금 유일한 존폐 항목이다.** §6.3이 지하철을 *"정시성 높음"* 한 줄로 단정했으나 **✅ 표시가 없다 — 검증된 적이 없다.** 버스를 기각한 논리가 지하철에도 겨눠지며, **B+·C 어느 쪽이든 이것이 무너지면 차별점 자체가 없다.**

### 9.1 그 다음 (조건부)

| 단계 | 조건 | 비고 |
|---|---|---|

| **C 재검토** | 위에서 "경로 선택에 반영"의 이득이 측정될 때만 | §7.2 리스크는 해소돼 있어 되살리는 비용은 낮다 |

> ### 버스 절벽 — 어디까지 되나
>
> | 구간 | 다음 차로 밀림 계산 |
> |---|---|
> | **지하철 (전 구간)** | ✅ **완전히 됨** — 시각표가 있다 |
> | 버스, 첫 정류장 ~3분 | ✅ 도착정보 (다음 2대) |
> | 버스, 첫 정류장 5~15분 | ⏳ **D0 수집 2주 뒤** — 위치정보 + 주행시간 모델 |
> | **버스, 중간 환승 — 시각표를 지키는 노선** | ⏳ **D0 수집의 부산물로 가능** ↓ |
> | **버스, 중간 환승 — 배차간격 운영** | ❌ 불가 — 지목할 차가 없다 |
>
> #### 공식 시각표가 없어도 이력으로 복원한다
>
> **노선마다 다르다:**
>
> | 노선 성격 | 이력이 주는 것 | 절벽 |
> |---|---|---|
> | **성긴 배차 · 광역/직행좌석** | **안정적 패턴** — "매일 08:05, 08:35, 09:05" | ✅ **복원 → 절벽** |
> | **도심 촘촘한 배차** (H<10분) | **분포만** — 오늘 08:30 차 ≠ 어제 08:30 차 | ❌ |
>
> #### ⚠️ 도착 시각이 아니라 **구간 소요시간**으로 재야 한다
>
> **도착 시각 이력을 그대로 쓰면 안 된다.** 분산이 섞이기 때문이다:
>
> ```
> 정류장 X 도착 시각의 분산  =  기점 출발 분산  +  주행시간 분산
> ```
>
> 이걸 뭉쳐 보면 **분산이 커도 원인을 모른다.** 판별이 안 된다:
>
> | 분해했을 때 | 뜻 | 절벽 |
> |---|---|---|
> | 출발 정시 · 주행 흔들림 | **기점 근처는 정확, 멀수록 부정확** | 기점 근처만 ✅ |
> | 출발부터 흔들림 | 어디서나 부정확 | ❌ |
>
> **구간 소요시간이 더 근본적인 이유 셋:**
>
> 1. **분해된다** — 위 표의 판별이 가능해진다. §8 #4 가 바로 이 구분이다
> 2. **실시간과 결합된다** — `지금 8번에 있다(위치정보) + 8→13 소요시간(이력) = 13번 도착`. 도착 시각 이력만으론 **"지금 그 차가 어디 있는지"를 못 쓴다.** §9.1 "지평 확장"이 이 결합이다
> 3. **일반화된다** — `A→B 6분` 은 그 노선의 **모든 차**에 적용된다. 도착 시각은 **특정 차 한 대**의 것이라 재사용이 안 된다
>
> **조합:**
> ```
> 기점 출발 시각 (시각표 복원 or 실시간)
>   + Σ 구간 소요시간 (이력 모델, 시간대·요일별)
>   = 어느 정류장이든 도착 시각        ← 중간 환승 정류장 포함
> ```
> 단 **기점 출발을 알아야** 하고, 그건 시각표를 지키는 노선에서만 된다. 그 한계는 그대로다.
>
> **✅ 수집기가 이미 구간 단위로 기록한다:**
> ```json
> {"t":"…", "t_prev":"…", "from_ord":31, "to_ord":32, "band":2, "daytype":"weekday", …}
> ```
> `from_ord → to_ord` 와 통과 시각 구간 `(t_prev, t]` — **도착 시각을 통째로 저장하는 게 아니라 구간으로 쪼개서** 넣는다. 그게 §4.4 의 `stop_times` 이고, 시각표 복원은 그 **부산물로 공짜로 나온다.** 별도 수집이 필요 없다. (신분당선을 `realtimePosition` 으로 복원하려던 것과 같은 발상 — §3.3.3)
>
> #### 배차간격 운영 노선에 이력이 주는 것 — 절벽이 아니라 CV
>
> **시각표는 몇 년을 모아도 안 나온다.** 존재하지 않기 때문이고, §5 가 버스를 `frequencies.txt`(`exact_times=0`) 로 매핑한 이유다. **C 로 가도 같다.**
>
> **대신 이력은 실제 `CV`(뭉침 정도)를 준다.** §6.3 이 지적한 *"CV 를 모르면 기대 대기가 10분 차이"* 를 메운다 → **절벽은 못 만들지만 바닥 오차를 줄인다.** §1 논리상 그것이 신호 90초를 측정 가능하게 만드는 조건이다.

**바뀐 것**

- **C 전체(0~8단계)가 로드맵에서 내려갔다.** B+ 는 KTDB·OTP·`stop_times` 가 전부 불필요 (§2·§4.4). 단 **수집은 백그라운드로 계속** — C 를 되살릴 선택지를 위해.
- **D0가 유일한 리드타임이다.** 나머지는 전부 코드라 순서를 바꿔도 되지만, **수집은 오늘 켜야 며칠 뒤에 답이 나온다.** 버스에서 이게 7개월이던 것이 지하철에선 며칠이다(§3.3.3).
- **판정 시점이 2주 뒤로 당겨졌다.** 이전 판은 5단계(수 개월 뒤)에서 가설을 검증했다.

### 의사결정 지점의 판정 기준 (수정)

이전 판: *"지하철 환승에서 **신호+계단** 보정이 3~5분씩 만들어내면 제품."* **셋을 고쳐야 한다.**

1. **신호와 계단을 묶지 말 것.** **계단은 역 안, 신호는 역 밖이다** — 지하철↔지하철 환승 통로에 횡단보도는 없다. 묶어서 3분이 나와도 계단 덕인지 신호 덕인지 모르고, 차별점으로 내세운 건 신호 쪽이다.
2. **베이스라인은 0이 아니다.** §1 가설은 "기존 길찾기는 신호 대기 0초"라지만 OTP는 이미 15초를 넣는다(§7.2 ③).
3. **크기만이 아니라 빈도를 함께 잴 것.** §6.3 "신호 × 절벽의 실제 영토" 표에서 차별점이 성립하는 조합은 **`버스/도보 → 도보 → 지하철`** 하나뿐이다. **차이가 커도 드물게 일어나면 제품이 안 된다.**

### 실현성 평가

| | 혼자 |
|---|---|
| 전국 GTFS 자체 조립 | ✅ **가능** — TAGO가 `cityCode` 루프 하나로 균일, ID는 접두사만 벗기면 됨 |
| 돌아가는 전국 길찾기 (기술) | 6개월~1년 |
| **믿을 수 있는 전국 서비스 (제품)** | ❌ 품질 책임·오류 대응 루프에 사람이 필요. 경쟁 상대는 **무료** |

**전국 GTFS 조립이 "250개 BIS 수작업"일 것이라는 우려는 실측으로 반증됐다** (§4.1, §4.2). 남은 건 조립 난이도가 아니라 **데이터 존재 여부**(서울 별도, 강원 등 구멍)와 **운영 책임**이다.

### 경쟁 관점

카카오는 **버스 통과 시각 이력**으로 "언제 오는가"를 정복했다(버스에 GPS가 달려 계속 쌓임). 그러나 **사용자의 도보 시간 이력**은 그렇게 쌓이지 않는다 — 특히 **지하 역사에서는 GPS가 잡히지 않는다.**

**GPS가 안 닿는 곳 = 이력이 안 쌓이는 곳 = 그들이 못 고치는 곳.** 지하철 환승 통로가 정확히 거기이고, 신호·계단은 여전히 모델로 풀어야 하는데 그들은 하지 않고 있다. "지하철 환승 시간이 말이 안 된다"는 체감이 이 구조에서 나온다.

다만 **평균으로 충분하다면 카카오가 이긴다**(이력이 더 두껍고 넓다). **지금 이 순간의 위상**이 의미가 있어야 차별점이 성립하고, 그것이 §9 의사결정 지점에서 검증할 대상이다.
