# 개인화 연결 예측 검증

목표는 간단하다.

> 같은 실제 이동에서 `네이버 원본 → 개인 속도 → 개인 속도+신호` 순으로
> 정류장 도착시각과 연결편 판정이 실제에 가까워지는가?

전체 앱·GTFS·그래프 빌드는 필요 없다. 사용자가 올린 지도 캡처와 스톱워치
타임라인을 연결 이벤트 단위 JSON으로 옮겨 `score.py`로 평가한다.

---

## 1. 가설과 통과 기준

고정한 모델을 **새로운 연결 이벤트 3건 이상**에 적용했을 때:

1. `Speed+Signal`의 정류장 도착 MAE가 네이버보다 **30% 이상 감소**
2. 실제 탑승 가능 연결편을 **3건 중 2건 이상 적중**
3. 잡을 수 있다고 했지만 실제로는 안전마진이 없었던 **위험한 오답 0건**
4. 네이버는 틀리고 제품만 맞은 **순증분 1건 이상**

이면 1차 통과다. 표본 3건 미만이면 결과가 좋아도 `판정 보류`다.

이 검증은 **해결책 성능**만 확인한다. 다른 사용자도 원하는지, 반복 사용하는지,
돈을 내는지는 별도의 사용자 수요 가설이다.

## 2. 비교 모델

| 모델 | 계산 | 역할 |
|---|---|---|
| `Naver` | 네이버 표시 도보시간 + 네이버 선택 연결 | 실제 경쟁 기준선 |
| `Speed` | 네이버 도보시간 × 개인 시간비율 | 속도 개인화의 순증분 |
| `Speed+Signal` | Speed + 첫 도보 실시간 위상 / 그 외 기대 신호대기 | 제품 예측 |
| `Oracle` | Speed + **실제 관측 신호대기** | 예측이 아닌 개선 가능 상한 |
| `TMAP-Personal` | TMAP 시설별 거리 ÷ 개인 물리속도 | 실제 해결책의 순수 이동 모델 |
| `TMAP+Signal` | TMAP-Personal + 횡단보도별 기대 신호대기 | TMAP 기반 제품 예측 |
| `TMAP-Oracle` | TMAP-Personal + **실제 관측 신호대기** | TMAP 모델의 개선 가능 상한 |
| `Kakao-Personal` | 카카오 도보거리 ÷ 개인 물리속도 | 장소명부터 자동화한 이동 모델 |
| `Kakao+Signal` | Kakao-Personal + 기대 신호대기 | 카카오 경로 기반 제품 예측 |
| `Kakao-Oracle` | Kakao-Personal + 실제 신호대기 | 카카오 모델의 개선 가능 상한 |

기본 개인 시간비율은 이전 무횡단 캘리브레이션의 `0.68`이다.
코드의 `speed_mps`는 `1.45배` 같은 비율이 아니라 실제 물리속도(m/s)이며,
신호의 통과 가능 창을 계산할 때만 사용한다.

신호 계산은 `docs/transit-routing-gtfs.md` §7의 규칙을 그대로 쓴다.

1. **첫 도보의 첫 횡단보도**이며 TMAP 횡단보도와 실시간 교차로가 50m 안에서
   매칭되고 예측 지평이 3분 이내면 실시간 상태·잔여시간을 고정 주기에 투영한다.
2. 같은 도보의 두 번째 횡단보도부터, 뒤쪽 도보, 실시간 미지원 지역,
   오래된 위상은 주기 기반 기대대기를 쓴다.
3. 어느 경우든 통과 조건은
   `녹색 잔여 ≥ 횡단거리 ÷ 사용자 속도`다.
4. 주기·녹색시간도 없으면 0초로 간주하지 않고 `Signal unavailable`로
   표시한다. 그 표본은 신호 모델 통과 판정에 넣지 않는다.

기대대기:

```text
crossing_time = crossing_distance / user_speed
effective_green = green - crossing_time
E[wait] = (red + crossing_time)² / (2 × cycle)
```

고정 주기, 균일 위상, `crossing_time ≤ green`이 전제다. 최악값은 기본 제품
예측이 아니라 민감도 비교에만 사용한다.

## 3. 관측 단위

전체 트립이 아니라 **도보 뒤 하나의 연결편을 타는 사건**이 표본 하나다.

예를 들어 7월 23일 이동은:

```text
E1  출발지 도보 → 6211
E2  6211 하차 후 도보 → G8110
E3  G8110 하차 후 도보 → 55
```

세 이벤트로 나눈다. 중간에 다른 버스를 선택해도 각 이벤트의 입력·정답이
분리되므로 전체 경로 변경이 도보 모델 평가를 오염시키지 않는다.

---

## 4. 현장에서 남길 것

### 도보 시작 시점

1. 휴대전화 절대시각
2. 네이버 경로 캡처
   - 도보거리
   - 도보시간
   - 네이버가 선택한 연결
3. 목표 정류장의 버스 도착정보 캡처
   - 후보 차량 ID/노선
   - 각 차량의 `N분 후`
   - 캡처 절대시각

### 이동 중

1. 실제 도보 시작시각
2. 신호별 대기 랩
3. 뛰기·정지·경로 이탈 여부

### 정류장 도착 이후

1. 실제 정류장 도착시각
2. 후보 차량들의 실제 출발시각
3. 실제 탑승 차량
4. 자발적으로 앞차를 보냈는지

`실제로 탄 차`와 `물리적으로 잡을 수 있었던 가장 이른 차`는 다르다.
스코어러의 정답은 실제 정류장 도착시각과 후보 차량 출발시각으로 계산한
**가장 이른 안전한 연결**이다. 실제 탑승 차량은 별도로 기록한다.

## 5. JSON 스키마

```json
{
  "id": "2026-07-25-E1",
  "source": "real",
  "split": "test",
  "valid": true,

  "walk_start": "2026-07-25T08:10:00",
  "profile": {
    "walk_time_ratio": 0.68,
    "speed_mps": 1.66
  },
  "naver": {
    "walk_distance_m": 781,
    "walk_time_s": 660,
    "selected_connection": "6211_next"
  },
  "tmap": {
    "start": {"lon": 127.0000000, "lat": 37.0000000, "name": "도보 시작"},
    "end": {"lon": 127.0000000, "lat": 37.0000000, "name": "정류장"},
    "facility_speed_multipliers": {
      "17": 0.875,
      "14": 0.775
    }
  },
  "kakao": {
    "start": {"query": "출발 장소명"},
    "trip_end": {"query": "전체 이동의 최종 목적지"},
    "prefer_vehicle": "6211",
    "prefer_boarding_stop": "실제 승차 정류장명",
    "route_mode": "SHORTEST"
  },
  "crossings": ["cw_home_1"],
  "signal": {
    "is_first_walk": true
  },
  "bus_snapshot": {
    "captured_at": "2026-07-25T08:09:55",
    "candidates": [
      {"id": "6211_early", "eta_s": 600},
      {"id": "6211_next", "eta_s": 1140}
    ]
  },
  "actual": {
    "stop_arrival": "2026-07-25T08:17:50",
    "signal_wait_s": 35.2,
    "connection_departures": [
      {"id": "6211_early", "departed_at": "2026-07-25T08:19:10"},
      {"id": "6211_next", "departed_at": "2026-07-25T08:28:40"}
    ],
    "boarded": "6211_early"
  }
}
```

`tmap`은 선택 필드다. 좌표는 WGS84 경도(`lon`)·위도(`lat`)이며, 실제 도보
구간의 시작점과 목표 정류장을 넣는다. 좌표가 없는 과거 표본은 기존
`Speed+Signal`까지만 계산된다. TMAP 원본 응답은 파일로 저장하지 않고 평가
실행 중 메모리에서만 사용한다.

`kakao`를 사용하면 장소명으로 좌표를 찾고 카카오 도보 경로의 전체 좌표열을
가져온다. 동명 장소가 있으면 `near`로 검색 중심과 반경을 제한하거나
`result_index`로 후보를 고정한다. `end`에 목표 정류장을 직접 넣을 수도 있고,
`trip_end`와 `prefer_vehicle`을 넣으면 카카오 대중교통 후보 중 해당 노선이
포함된 경로의 최초 승차점 좌표를 자동 추출한다. 관측 검증에서는
`prefer_boarding_stop`도 함께 넣어 실제로 걸어간 정류장이 아닌 후보가
선택되는 것을 막는다. 후보에 없으면 자동 추정을 강행하지 않고 오류로 중단한다.
아파트 출입구처럼 장소 대표점과 실제 시작점이 다른 경우에는 `start`에
휴대전화가 기록한 `lon`·`lat`를 직접 넣는다. `--hybrid`에서는 동일한
시작·종료 좌표를 TMAP에 넘겨 `facilityType=15` 횡단보도 구간을 검출한다.
TMAP은 신호 주기·녹색 시간·현재 위상을 제공하지 않으므로 그 값은
`crosswalks.json` 또는 실제 관측으로 계속 보완한다.

### `source`와 `split`

| 값 | 의미 | 통과 판정 포함 |
|---|---|---|
| `source: demo` | 인위적으로 만든 예시 | 제외 |
| `split: calibration` | 이미 결과를 보고 모델을 만든 과거 자료 | 제외 |
| `source: real`, `split: test` | 모델을 고정한 뒤 얻은 새 관측 | 포함 |

7월 23일 E1은 입력 파이프라인 확인을 위한 `calibration` 표본으로 이미 들어 있다.
새 실험에서는 모델 비율·신호식을 바꾸지 않고 `test`만 추가한다.

## 6. 횡단보도 데이터

`crosswalks.json`:

```json
{
  "cw_home_1": {
    "source": "real",
    "dist_m": 18,
    "cycle_s": 95,
    "green_s": 20,
    "stdg_cd": "11140",
    "crsrd_id": "교차로ID",
    "direction": "et"
  }
}
```

- `dist_m`: 연석부터 반대 연석까지의 횡단거리
- `cycle_s`: 녹색 시작부터 다음 녹색 시작까지
- `green_s`: 진입 가능한 보행 녹색시간
- `stdg_cd`·`crsrd_id`: 실시간 API 교차로 식별자
- `direction`: `nt/ne/et/se/st/sw/wt/nw`. TMAP 선형으로 자동 추정하지만
  실제 API 방향과 다르면 이 값으로 고정한다.
- 가능하면 3주기 평균
- 감응식이면 메모하고 별도 표본으로 표시

실시간 API는 현재 상태와 잔여시간만 주며 `cycle_s`와 `green_s` 필드는 없다.
주기는 전국횡단보도표준데이터의 제공값을 우선 쓰고, 없으면 같은
`crsrd_id + direction`을 1~2초 간격으로 폴링해 녹색 시작 간격을 2회 이상
관측한다. `signal_client.infer_timing()`은 그 전이 기록을 주기와 녹색시간으로
변환한다. 성남·판교처럼 실시간 API가 없는 지역도 주기만 확보하면 기대대기
분기로 평가할 수 있다.

횡단거리 이동시간 자체는 네이버 도보시간에 이미 포함된 것으로 본다.
신호 모델은 **추가 대기시간만** 더한다.

주기 데이터가 아직 없으면 `crossings: []`로 Speed 모델부터 검증한다.
실제 신호대기는 `Oracle`에만 들어가므로 기록은 계속 남긴다.

## 7. 실행

```bash
cd spike/validation

# 캘리브레이션 포함 전체 확인
python3 score.py

# 새 테스트 표본만
python3 score.py --split test

# collector/.env의 TMAP_APP_KEY로 좌표가 있는 표본 조회
python3 score.py --tmap
python3 score.py --tmap --split test

# 장소명→좌표→카카오 도보 경로
python3 score.py --kakao

# 카카오 자동 좌표/경로 + TMAP 시설 유형
python3 score.py --hybrid

# TMAP 횡단보도 + 실시간 보행신호. 실시간 불가 시 주기 기반 기대값
python3 score.py --hybrid --signals

# 안전마진 민감도
python3 score.py --buffer 30
python3 score.py --buffer 60
python3 score.py --buffer 90

# 개인 시간비율 민감도(판정 확정용이 아닌 진단)
python3 score.py --ratio 0.65
python3 score.py --ratio 0.72

# 실측 횡단보도의 속도별 유효 녹색 변화
python3 score.py --sweep

# 단위 테스트
python3 -m unittest test_score.py
```

기본 안전마진은 60초다. 실제 버스가 최초 화면보다 30~90초 당겨지는 관측을
반영한 값이며, 30/60/90초 결과를 함께 보고 민감도를 확인한다.

## 8. 결과 읽기

각 이벤트에서 출력하는 값:

- 모델별 정류장 도착 예상시각
- 실제 대비 도착 오차
- 모델이 선택한 가장 이른 연결
- 실제로 잡을 수 있었던 가장 이른 연결
- 연결 적중 여부
- 위험한 오답 여부

요약:

- MAE: 정류장 도착시각 평균 절대오차
- 중앙 AE: 소표본에서 극단값 영향을 줄인 오차
- 연결정확도
- 위험오답 수
- 순증분: 네이버는 틀리고 제품만 맞은 표본

`Oracle`은 실제 신호대기를 입력하므로 제품 정확도로 세지 않는다.
`Speed`와 `Oracle` 차이가 클 때만 신호 데이터 확보의 가치가 크다고 해석한다.

## 9. 제외 규칙

다음은 `valid: false`로 남기고 통과 판정에서는 제외한다.

- 달림 또는 무단횡단
- 개인 용무로 정지
- 경로 이탈
- 버스 도착 캡처의 절대시각을 모름
- 후보 차량의 실제 출발을 확인할 수 없음
- 무배차로 계획 자체가 붕괴

경로 변경 자체는 제외 사유가 아니다. 연결 이벤트를 나누고 변경 이유와 실제 후보
차량 출발을 기록하면 별도의 유효 표본으로 사용할 수 있다.
