# 개인화 연결 안내 — 1~2일 경량 해결책 검증

## 0. 문서 목적

이 실험은 길찾기 앱이나 신호 데이터 시스템을 개발하는 실험이 아니다.

검증하려는 것은 다음 해결책의 가장 작은 형태다.

> 네이버가 표시한 도보시간을 사용자의 실제 보행속도로 보정하면,
> 기존 안내보다 실제로 잡을 수 있는 가장 이른 연결편을 더 정확하게 판단할 수
> 있는가?

이를 검증하기 위해 앱 대신 사람이 계산해주는 **Concierge MVP**를 사용한다.
사용자는 평소처럼 네이버지도를 켜고 이동하며, 평가자는 미리 고정한 계산식으로
개인화 연결 안내를 만든다.

이 문서는 다음을 고정한다.

- 무엇을 가설로 검증하는가
- 어떤 이동을 몇 번 관측하는가
- 이동 전후에 어떤 자료를 남기는가
- 개인화 안내를 어떤 식으로 계산하는가
- 무엇을 성공과 실패로 판단하는가
- 결과를 본 뒤 기준을 바꾸는 것을 어떻게 막는가

---

## 1. 이번 검증의 범위

### 1.1 검증하는 것

- 사용자의 개인 보행시간을 반영한 정류장 도착 예측
- 개인화 예측이 선택한 가장 이른 연결편
- 네이버 안내 대비 도착 오차 감소
- 네이버가 다음 차를 안내했지만 개인화 계산은 이른 차를 잡는다고 판단하는
  **순증분 사례**
- 신호 대기가 연결 판단을 뒤집는 임계조건

### 1.2 검증하지 않는 것

- TMAP API 연동
- 카카오 API 연동
- 실시간 보행신호 API 연동
- 횡단보도 좌표 자동 매칭
- 신호 주기 자동 학습
- 완성된 앱의 사용성
- 다른 사용자에게도 같은 문제가 발생하는지
- 결제 의향이나 시장 규모

API와 자동화는 이 경량 실험이 통과한 뒤 진행할 **기술 실현성 검증**이다.

### 1.3 이전 실험과의 차이

이전 실험은 다음 사실을 확인한 **문제 검증**이었다.

- 사용자가 네이버의 평균 보행자보다 약 1.4~1.5배 빠르다.
- 긴 도보에서는 앱보다 이른 연결편을 실제로 잡는 사례가 있다.

이번 실험은 이미 확인한 속도 차이를 실제 안내에 사용했을 때 연결 판단이
개선되는지 보는 **해결책 검증**이다.

## 2. 가설

### 2.1 핵심 가설

> 출발 전에 네이버 도보시간을 개인 시간비율 `0.68`로 보정하여 연결편을
> 판정하면, 새로운 이동 3회 중 2회 이상에서 실제로 잡을 수 있는 가장 이른
> 연결편을 정확히 맞히고, 잡을 수 없는 차를 잡을 수 있다고 안내하는
> 위험한 오답은 0회일 것이다.

### 2.2 보조 가설

> 개인화 도착시각의 절대오차는 네이버 도착시각의 절대오차보다 평균 30% 이상
> 작을 것이다.

### 2.3 신호에 관한 탐색 질문

이번 실험에서는 신호를 사전에 정확히 예측하지 않는다. 대신 다음을 관찰한다.

- 이른 연결편을 잡기 위해 허용되는 최대 신호 대기시간은 얼마였는가?
- 실제 신호 대기시간이 그 임계값 안에 들었는가?
- 순수 보행속도에서 얻은 시간이 신호 대기로 얼마나 흡수되었는가?
- 개인화 안내가 틀렸다면 원인은 순수 보행시간 오차인가, 신호 대기인가?

신호 결과는 핵심 가설을 설명하는 진단값이며, 실시간 신호 예측 성능으로
해석하지 않는다.

## 3. 실험 설계

### 3.1 관측 횟수

새로운 전체 이동을 **3회** 관측한다.

기존 2026년 7월 23일 이동은 모델 비율을 정한 `calibration` 자료이므로 최종
통과 판정에 포함하지 않는다.

가능하면 유효 표본 손실에 대비하여 4회 이동하고, 제외 기준에 걸리지 않은
앞의 3회를 판정에 사용하는 것이 안전하다.

### 3.2 세 가지 Case

| Case | 시간 | 물리 경로 | 목적 |
|---|---|---|---|
| 1 | calibration과 비슷한 평일 시간대 | 동일 | 같은 조건에서 재현되는지 |
| 2 | 다른 시간대 | 동일 | 배차·신호 조건이 달라도 동작하는지 |
| 3 | 다른 시간대 | 일부 또는 전체가 다름 | 다른 도보·연결 구조로 확장되는지 |

Case 1의 권장 시간 범위는 calibration 출발시각의 약 ±20~30분이다. 요일이
완전히 같지 않아도 평일과 비슷한 시간대라면 진행할 수 있지만, 결과에 요일
차이를 기록한다.

### 3.3 같은 경로의 정의

버스 번호가 아니라 **물리적 이동 구조**로 판단한다.

다음이 같으면 같은 경로다.

- 도보 동선
- 승차 정류장
- 하차 정류장
- 환승 정류장
- 횡단보도

예를 들어 1150·9000·9300이 같은 정류장에서 출발해 같은 정류장에 내리고
동일 구간을 통과한다면 같은 연결 그룹으로 볼 수 있다.

다음 중 하나가 달라지면 다른 경로다.

- 승하차 정류장이 다름
- 반대편 정류장이나 다른 승강장으로 이동
- 도보 동선이나 횡단보도가 달라짐
- 환승 장소가 달라짐

순천향대학병원까지는 같고 이후에 다른 정류장에서 내려 다른 방식으로
이동하는 경로는 Case 3으로 사용할 수 있다.

### 3.4 보행 조건

세 번 모두 다음 규칙을 지킨다.

- 평소의 빠른 자연 보행
- 뛰지 않음
- 무단횡단하지 않음
- 초록 진입이 애매하면 다음 신호를 기다림
- 개인 용무로 정지하지 않음
- 의도적으로 경로를 단축하지 않음

불가피하게 규칙을 어겼다면 숨기지 않고 특이사항에 기록한다.

## 4. 평가 단위

### 4.1 연결 이벤트

기본 평가 단위는 하나의 **도보 → 연결편 탑승 사건**이다.

예:

```text
E1  출발지 도보 → 6211
E2  6211 하차 후 도보 → G8110
E3  G8110 하차 후 도보 → 55
```

한 번의 전체 이동에 여러 연결 이벤트가 포함될 수 있다.

### 4.2 핵심 판정과 보조 분석

- 전체 이동 3회를 서로 다른 **Case**로 본다.
- 각 Case에서 출발 전에 자료가 확보된 첫 번째 연결 이벤트를 핵심 판정에 쓴다.
- 중간 환승 이벤트도 필요한 캡처가 있으면 같은 방식으로 계산하지만 보조 분석으로
  표시한다.
- 한 번의 이동에서 나온 E1·E2·E3을 서로 완전히 독립된 3회 실험처럼 부풀리지
  않는다.

## 5. 실험 전에 고정하는 값

결과를 본 뒤 아래 값을 바꾸지 않는다.

| 항목 | 고정값 |
|---|---:|
| 개인 시간비율 | `0.68` |
| 개인 물리속도 참고값 | `1.66m/s` |
| 기본 예측 안전마진 | `60초` |
| 민감도 안전마진 | `30초`, `90초` |
| 최소 전체 이동 | `3회` |
| 연결 적중 통과 기준 | `3회 중 2회 이상` |
| 위험한 오답 허용 | `0회` |
| 도착 MAE 개선 기준 | `30% 이상` |
| 순증분 기준 | `1회 이상` |

`0.68`은 기존 무횡단 구간 calibration에서 정한 값이다. 새 결과가 좋지 않다는
이유로 0.65나 0.72로 다시 맞추면 검증이 아니라 사후 최적화가 된다.

## 6. 이동 전 자료

이동 전 자료는 실제 도보 결과를 보기 전에 캡처한다. 평가 계산은 나중에 해도
되지만 입력 화면의 캡처 시각이 이동 전이어야 한다.

### 6.1 이동 전 텍스트

```text
[이동 전]

날짜:
현재 시각:
예정 출발시각:
출발지:
도착지:
이번 Case: 1 / 2 / 3
평일/주말:
예정한 물리 경로:
특이사항:
```

### 6.2 필수 캡처

1. **네이버 추천 경로 전체**
   - 검색 기준시각
   - 예상 출발·도착시각
   - 총 예상시간
   - 네이버가 선택한 노선과 환승

2. **첫 도보 구간**
   - 도보거리
   - 네이버 도보 예상시간
   - 목표 승차 정류장 이름

3. **도보 시작 직전 버스 도착정보**
   - 캡처 절대시각
   - 후보 노선
   - 첫 번째 후보와 다음 후보의 `N분 후`
   - 가능하면 남은 정류장 수

### 6.3 권장 캡처

- 같은 구간을 운행하는 대체 버스 목록
- 네이버가 선택하지 않은 이른 후보
- 경로 상세의 각 도보 구간
- 휴대전화 상단 시계가 함께 보이는 화면

### 6.4 캡처 시각 규칙

버스 도착정보는 시간이 계속 바뀌므로 화면 안에 휴대전화 시계가 보이거나
파일 생성시각으로 캡처 절대시각을 확인할 수 있어야 한다.

`10분 후`라는 글자만 있고 언제 캡처했는지 모르면 절대 출발시각을 복원할 수
없으므로 해당 연결 이벤트는 제외한다.

## 7. 이동 전에 만드는 개인화 안내

평가자는 이동 전 입력만 사용하여 다음을 계산한다. 실시간으로 답을 받지
못하더라도 계산식이 고정되어 있으므로, 이동 전 캡처를 나중에 같은 식으로
계산할 수 있다.

### 7.1 개인화 순수 도보시간

```text
개인화 순수 도보시간
= 네이버 도보 예상시간 × 0.68
```

예:

```text
네이버 도보시간 = 11분 = 660초
개인화 순수 도보시간 = 660 × 0.68 = 448.8초
                       = 7분 28.8초
```

### 7.2 후보 버스 예상 출발시각

```text
후보 버스 예상 출발시각
= 버스정보 캡처시각 + 화면의 도착 예정시간
```

예:

```text
캡처시각 = 15:22:00
첫 후보 = 10분 후
예상 출발시각 = 15:32:00
```

화면의 분 단위 ETA에는 반올림 오차가 있으므로 기본 결과 외에 안전마진
30·60·90초를 함께 계산한다.

### 7.3 개인화 정류장 도착시각

```text
개인화 도착시각
= 실제 도보 시작시각 + 개인화 순수 도보시간
```

사전 계산 때 실제 도보 시작시각이 아직 없으면 예정 출발시각을 사용하고,
최종 채점 때 스톱워치의 실제 시작시각으로 다시 계산한다. 시간비율이나 기준은
바꾸지 않는다.

### 7.4 연결편 선택

기본 안전마진 60초를 적용한다.

```text
후보 버스를 잡을 수 있다고 예측
⟺ 후보 예상 출발시각 ≥ 개인화 도착시각 + 60초
```

조건을 만족하는 후보 중 가장 이른 연결편을 개인화 모델의 선택으로 기록한다.

### 7.5 허용 가능한 최대 신호 대기시간

```text
최대 허용 신호 대기
= 후보 예상 출발시각
 - 실제 도보 시작시각
 - 개인화 순수 도보시간
 - 안전마진
```

예:

```text
도보 시작부터 버스까지 남은 시간 = 10분
개인화 순수 도보시간            = 7분 29초
안전마진                         = 1분

최대 허용 신호 대기 = 10:00 - 7:29 - 1:00
                     = 1분 31초
```

안내문은 다음 형태로 고정한다.

```text
개인화 예상 도착: 15:29:29
예측한 가장 이른 연결: 6211 첫 후보
허용 가능한 총 신호 대기: 1분 31초
이보다 신호가 길면 다음 6211을 타야 함
```

이 값은 신호 예측이 아니라 연결 판단이 뒤집히는 **임계값**이다.

## 8. 이동 중 기록

### 8.1 스톱워치 원칙

- 전체 이동 시작과 동시에 스톱워치 시작
- 상태가 바뀔 때마다 랩
- 도보, 신호 대기, 버스 대기, 버스 이동을 분리
- 신호가 여러 개면 각각 별도 랩
- 신호에서 기다리지 않았다면 `00:00.00`으로 기록

### 8.2 기존 기록 형식

기존 형식을 그대로 사용한다.

```text
7/23(15:22 출발 → 16:29 도착)

도보(07:52.81): 신호 대기(03.54)
버스 대기(01:04.39)
버스 이동(13:47.21): 6211번 버스 이동
도보(27.21)
버스 대기(28.63)
버스 이동(17:32.72): G8110번 이동
도보(05:38.23): 보행신호 대기(45.52, 02:00.77)
버스 대기(52.60)
버스 이동(18:14.33): 55번 이동
도보(48.09)
```

원본 랩 목록도 함께 보낸다.

```text
1      06:56.08      06:56.08
2      00:03.54      06:59.62
3      00:53.19      07:52.81
...
```

### 8.3 도보 구간 분해

```text
실제 총 도보시간
= 순수 걷기 + 신호 대기

실제 순수 걷기
= 실제 총 도보시간 - 신호 대기 합계
```

이를 통해 다음 두 오차를 분리한다.

- 개인 속도 모델의 순수 걷기 오차
- 신호 때문에 추가된 지연

### 8.4 특이사항

다음이 발생하면 반드시 기록한다.

- 달림
- 무단횡단
- 개인 용무로 정지
- 길을 잘못 듦
- 네이버 경로에서 이탈
- 승객 혼잡으로 승차 실패
- 버스가 정류장을 무정차 통과
- 버스 ETA가 갑자기 크게 변함
- 일부러 앞차를 보냄
- 이동 중 추천 경로가 변경됨

---

## 9. 정류장 도착과 탑승 자료

### 9.1 필수 캡처

각 핵심 연결 이벤트에서 다음 화면을 남긴다.

1. 도보 종료 후 정류장 도착 직전 또는 직후의 버스정보
2. 실제 버스 탑승 직전의 버스정보
3. 가능하면 탑승한 버스 번호가 보이는 화면

### 9.2 필수 시각

- 실제 정류장 도착시각
- 실제 버스 탑승시각
- 정류장에서 기다린 시간
- 탑승한 노선

### 9.3 실제로 잡을 수 있었던 가장 이른 연결

정답은 단순히 네이버가 안내한 버스도 아니고, 사용자가 선택해 탄 버스도 아니다.

> 사용자가 실제 정류장에 도착한 뒤 물리적으로 탈 수 있었던 가장 이른 후보

를 정답으로 한다.

실제로 탄 버스는 반드시 잡을 수 있었던 버스로 인정한다. 사용자가 자발적으로
앞차를 보냈다면 앞차의 출발 증거를 확인해 정답을 별도로 판정한다.

예측 안전마진 60초는 제품이 보수적으로 안내하기 위한 기준이다. 실제 정답을
판정할 때 사용자가 도착 10초 뒤 버스를 실제로 탔다면 그 버스는 잡을 수 있었던
버스다. 안전마진 때문에 실제 탑승 성공을 실패로 바꾸지 않는다.

## 11. 계산 지표

### 11.1 도착 오차

```text
네이버 도착오차
= 네이버 예상 정류장 도착시각 - 실제 정류장 도착시각

개인화 도착오차
= 개인화 예상 정류장 도착시각 - 실제 정류장 도착시각

절대오차
= |도착오차|
```

### 11.2 평균 절대오차

```text
MAE = 각 Case 절대오차의 평균
```

개선율:

```text
MAE 개선율
= 1 - (개인화 MAE ÷ 네이버 MAE)
```

### 11.3 연결 적중

```text
연결 적중
⟺ 개인화 모델이 선택한 가장 이른 연결
   = 실제로 잡을 수 있었던 가장 이른 연결
```

### 11.4 위험한 오답

다음이면 위험한 오답이다.

> 개인화 모델은 해당 버스를 잡을 수 있다고 안내했지만, 사용자가 규칙대로
> 이동했음에도 정류장 도착 전에 그 버스가 출발한 경우

다음은 모델의 위험한 오답으로 세지 않고 별도 제외·주석 처리한다.

- 버스 무정차
- 승차 거부
- 사고나 돌발 통제
- 사용자의 달림·정지·경로 이탈
- 캡처 시각을 확인할 수 없음

### 11.5 순증분

```text
순증분
⟺ 네이버가 선택한 연결은 실제 최초 연결과 다르고,
   개인화 모델만 실제 최초 연결을 맞힘
```

### 11.6 신호 흡수량

```text
순수 속도 이득
= 네이버 도보시간 - 실제 순수 걷기

신호 흡수량
= 실제 신호 대기

실현된 도보 이득
= 순수 속도 이득 - 신호 흡수량
```

신호 대기가 순수 속도 이득보다 크면 빠른 보행의 장점이 해당 이벤트에서는
신호에 의해 모두 흡수된 것으로 해석한다.

## 12. 통과 기준

다음 네 조건을 모두 만족하면 1차 통과다.

1. 새로운 전체 이동 3회 중 개인화 연결 적중 **2회 이상**
2. 위험한 오답 **0회**
3. 개인화 도착 MAE가 네이버보다 **30% 이상 감소**
4. 네이버는 틀리고 개인화 모델만 맞힌 순증분 **1회 이상**

### 12.1 판정 보류

다음이면 통과나 실패로 단정하지 않고 보류한다.

- 유효한 전체 이동이 3회 미만
- 버스 출발 증거가 없어 실제 최초 연결을 판단할 수 없음
- 세 Case 모두 연결 후보 사이의 간격이 너무 커서 두 모델이 항상 같은 버스를
  선택함
- 모든 Case의 도보가 너무 짧아 개인 속도 차이가 연결 판단에 영향을 줄 수 없음

### 12.2 실패 해석

실패했다고 바로 서비스 전체를 폐기하지 않는다. 원인에 따라 다음 결정을 한다.

| 실패 원인 | 해석 | 다음 행동 |
|---|---|---|
| 순수 걷기 예측부터 틀림 | 0.68이 경로에 일반화되지 않음 | 경사·계단·거리별 속도 재검토 |
| 순수 걷기는 맞고 신호 때문에 틀림 | 신호 정보가 제품에 필수 | 신호 예측 기술 실현성 검증 |
| 버스 ETA가 크게 변함 | 도보보다 버스 예측 오차가 큼 | 연결 판정의 불확실성 범위 표시 |
| 늘 같은 연결을 선택 | 실험 Case가 약함 | 긴 도보+중간 배차 Case 추가 |
| 개인화가 더 정확하지만 순증분 없음 | 정확도 개선은 있으나 행동 차이 없음 | 더 절벽에 가까운 Case 관측 |

---

## 13. 제외 기준

다음 이벤트는 원자료에는 남기되 핵심 통과 판정에서 제외한다.

- 달림
- 무단횡단
- 개인 용무로 정지
- 경로 이탈
- 캡처 절대시각 누락
- 첫 후보·다음 후보를 식별할 수 없음
- 실제 정류장 도착시각 누락
- 실제 버스 출발 또는 탑승시각을 판단할 수 없음
- 무배차·운행중단으로 계획 자체가 붕괴

경로가 바뀌었다는 사실만으로 제외하지 않는다. 변경 전후 자료와 이유가
남아 있으면 별도 이벤트 또는 Case 3으로 분석할 수 있다.

제외 기준은 결과를 본 뒤 유리한 표본만 남기는 데 사용하면 안 된다.

## 14. 결과 누수 방지

### 14.1 반드시 결과 전에 고정할 것

- 개인 시간비율 0.68
- 안전마진 60초
- Case 구분
- 네이버 도보시간
- 버스정보 캡처시각과 후보 ETA
- 제외 기준
- 통과 기준

### 14.2 결과 후에 하면 안 되는 것

- 실제 도착에 맞게 시간비율 변경
- 실제 탄 버스를 모델의 추천으로 바꿈
- 실패 Case만 “경로가 달랐다”며 제거
- 실제 신호 대기를 예측값처럼 입력
- 안전마진을 Case마다 다르게 선택
- 30·60·90초 중 결과가 가장 좋은 값만 공식 결과로 채택

30초와 90초는 민감도 분석일 뿐이고 공식 판정은 60초를 사용한다.

### 15.3 결론 문장

통과 예시:

```text
개인 시간비율 0.68을 사용한 수동 연결 안내는 새로운 전체 이동 3회 중
2회에서 실제 최초 연결을 맞혔고 위험한 오답은 없었다. 개인화 도착 MAE는
네이버보다 35% 감소했으며, 네이버는 다음 차를 안내했지만 개인화 안내만
이른 차를 맞힌 순증분이 1회 있었다. 따라서 개인 속도를 반영한 연결 안내는
1차 해결책 가설을 통과했다.
```

조건부 또는 실패 예시:

```text
순수 걷기 예측은 네이버보다 정확했지만 신호 대기가 큰 Case에서 이른 버스를
잡을 수 있다고 잘못 판단했다. 따라서 개인 속도 보정의 가치는 확인했으나,
안전한 연결 안내를 위해서는 신호 대기 예측 또는 보수적 불확실성 표시가
필요하다.
```

## 16. 실행 순서 요약

1. 세 Case와 고정값을 확인한다.
2. 이동 전에 네이버 경로와 버스 도착정보를 캡처한다.
3. 고정식으로 개인화 시간·연결편·신호 임계값을 계산한다.
4. 평소의 빠른 자연 보행으로 이동한다.
5. 도보·신호·버스 대기·버스 이동을 랩으로 분리한다.
6. 정류장 도착과 탑승 직전 버스정보를 캡처한다.
7. 이동 후 텍스트·원본 랩·캡처를 함께 제출한다.
8. 세 Case의 네이버·개인화·실제를 동일 기준으로 비교한다.
9. 사전에 정한 네 조건으로 통과·실패·보류를 결정한다.

이 실험이 통과한 뒤에만 TMAP 횡단보도, 공공데이터 신호, 주기 학습을 연결한
기술 MVP로 넘어간다.
