# bus-store

**도보 + 대중교통 길찾기** — 네이버·카카오지도의 길찾기와 같은 성격이되 **자동차는 제외**한다.

**차별점:** 걷는 속도(사용자 설정)와 **보행신호 대기**를 소요시간에 반영한다. 기존 서비스가 하지 않는 부분이다.

**가설:** 기존 길찾기는 도보 구간에 신호 대기가 0초로 들어가 있어 소요시간이 체계적으로 낙관적이다.
근거(✅ 실측) — TMAP 보행자 응답의 구간별 속도를 역산하면 **횡단보도(1.52 m/s)가 일반보행로(1.20 m/s)보다 빠르다.** 신호 대기가 조금이라도 반영됐다면 훨씬 느려야 한다. 즉 대기 = 0초.

## 빠른 시작 (수집)

```bash
cd collector
cp .env.example .env && chmod 600 .env   # 키 채우기 — 발급처는 .env.example 참조
python3 fetch_routes.py                   # 노선 풀 (경기 2,200노선 + 운행시간, ~10분)
python3 server.py                         # 수집 + 대시보드 → http://localhost:877 (리눅스 서비스는 8080)
```

⚠️ **항상 켜진 기계에서 돌려야 한다.** 컴퓨터가 꺼진 시간대의 데이터는 며칠을 기다려도 **0**이다 — 무작위 손실이 아니라 구조적 구멍이고, 하필 **출근·퇴근 첨두**가 노트북이 덮여 있는 시간이다. [`collector-design.md` §1.2](docs/collector-design.md)

## 실측으로 뒤집힌 것들

문서의 ✅ 는 전부 실호출로 확인한 것이다. 추정과 실물이 자주 달랐다:

| 통설 / 이전 판 | 실물 |
|---|---|
| API 제약은 **"30 TPS"** | **동시 세션 30 + 토큰 rate limit — 별개의 두 제약** |
| 실패하면 에러가 온다 | **세션 초과가 HTTP 200 + `resultCode 99`** 로 온다 → `resultCode` 를 안 보면 **조용히 데이터를 버린다** (실제로 요청의 2/3를 잃었다) |
| rate 만 지키면 된다 | **버스트가 문제다.** 균등 6.0/s = 429 0건 vs **같은 6.7/s 를 버스트로 = 78% 실패** |
| 위치정보 = 버스 GPS | **정류장 단위다.** 좌표가 버스가 아니라 **현재 정류장의 좌표**이고 정류장을 넘을 때만 바뀐다 |
| 평균 응답 500ms (활용가이드) | **~3초** |
| 갱신 일 1회 | **실시간(10~20초)** — 포털 표기가 맞았다 |
| 서울 버스 375노선 | **665개** |
| 서울도 TAGO 지원 | **TAGO 138개 도시에 서울만 없다** |
| 신분당선 데이터 없음 | **운영사 공식 PDF** + `realtimePosition`(판교 포함) |
